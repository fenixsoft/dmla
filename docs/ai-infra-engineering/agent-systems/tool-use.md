# 工具调用

LLM 只能生成文本，无法直接操作外部世界。工具调用（Tool Use）是弥补这一能力鸿沟的机制：模型生成结构化的调用指令，由执行引擎将其转化为实际的 API 请求、数据库查询、文件操作等动作，再将结果返回模型。这篇文章回答的核心问题是：如何设计一个让 LLM 可靠地调用外部工具的系统？

## 工具调用的原理

### 从文本生成到函数调用

- LLM 工具调用的本质是一种结构化输出：模型不再生成自由形式的文本，而是生成符合特定 schema 的 JSON 对象，其中包含函数名和参数
- 工具调用并非 LLM 的"新能力"，而是将文本生成能力导向结构化输出的结果。模型通过训练学会在特定提示格式下输出工具调用格式，而非自由文本
- 2023 年，OpenAI 在 GPT-4 中首次引入了函数调用（Function Calling）功能，模型可以输出标准化的函数调用 JSON，由客户端解析执行
- 工具调用与普通文本生成的关键区别：普通生成追求流畅自然，工具调用追求精确解析。一个多余的逗号可能导致 JSON 解析失败

### 工具描述的工程

- 工具描述（Tool Description）是告诉模型"有哪些工具可用、每个工具做什么、参数是什么"的信息，通常以 JSON Schema 或自然语言描述的形式注入提示词
- 描述的质量直接影响模型选择工具和填写参数的准确率。模糊的描述导致错误调用，冗长的描述浪费上下文空间
- 好的工具描述应包含：功能摘要（一句话说明工具做什么）、参数说明（每个参数的类型、含义、取值范围）、使用示例（典型调用场景）
- 工具描述的工程化实践：使用一致的命名规范、提供枚举值约束、标注必选与可选参数、说明副作用与权限需求

### 工具选择的决策过程

- 模型选择工具的过程是隐式的推理，模型根据用户意图和工具描述，在可用工具中选择最合适的一个
- 影响工具选择准确率的因素：工具数量（选择过多时准确率下降）、工具描述的区分度（功能相似的工具容易混淆）、用户意图的明确程度
- 减少选择错误的策略：按功能域分组工具（先选域再选工具）、提供工具使用示例、在描述中明确适用场景和不适用场景
- 工具选择错误是最常见的失败模式之一，因为模型倾向于选择描述中关键词与用户查询匹配度最高的工具，而非真正合适的工具

## 工具调用的协议设计

### 函数调用格式

- 主流函数调用格式：OpenAI Function Calling（JSON Schema 描述 + 工具调用对象）、Anthropic Tool Use（类似但支持更丰富的输入模式）、开源模型的原生工具格式（如 Qwen、Llama 的 chat template 内置工具格式）
- 格式的核心要素：工具名称（字符串）、参数对象（键值对）、调用标识（用于匹配调用与返回结果）
- 格式设计的关键决策：参数类型的丰富度（仅支持字符串 vs 支持数值、数组、嵌套对象）、可选参数的处理方式、错误信息的传递格式
- 不同格式之间的兼容性是工程实践中的常见痛点，抽象层（如将不同模型的工具格式统一为内部表示）是必要的

### 并行调用与依赖处理

- 模型在一次推理中可能生成多个工具调用，这些调用之间可能存在依赖关系（调用 B 依赖调用 A 的结果）或相互独立
- 并行调用（Parallel Tool Calls）的前提是调用之间无依赖关系，独立调用可以同时执行以减少总延迟
- 依赖关系的识别：显式依赖（参数引用前一次调用的返回值）容易识别，隐式依赖（两次调用共享同一副作用，如都修改同一文件）容易遗漏
- 依赖处理策略：保守策略（顺序执行所有调用，安全但慢）、激进策略（默认并行，仅处理显式依赖）、显式标注（由工具描述声明依赖关系）

### 流式工具调用

- 长时间运行的工具（如 Web 搜索、数据库查询）可能导致模型等待时间过长，流式工具调用允许工具在执行过程中增量返回结果
- 流式调用的挑战：模型需要决定是等待完整结果还是基于部分结果继续推理；部分结果可能导致过早的决策
- 流式调用与普通流式输出的区别：普通流式输出是 token 级别的增量，工具流式调用是结果级别的增量（如搜索结果逐条返回）

## 工具的实现模式

### 内置工具与自定义工具

- 内置工具是平台预定义的通用工具，如代码执行器、Web 搜索、文件读写；自定义工具是用户为特定场景开发的专用工具
- 内置工具的优势是经过充分测试和优化，劣势是灵活性有限；自定义工具的优势是精确匹配需求，劣势是开发成本和可靠性风险
- 自定义工具的开发最佳实践：输入验证（防止模型传入非法参数）、错误处理（返回结构化的错误信息而非异常）、幂等性（重复调用不应产生副作用）、超时控制（防止工具永远不返回）
- 工具的安全沙箱：代码执行工具应在隔离环境中运行，防止模型生成的代码影响宿主系统

### 工具的权限与安全

- 工具调用是 LLM 影响外部世界的唯一通道，因此工具的安全设计是 Agent 安全的核心
- 权限模型：每个工具声明其所需的权限级别（只读、写入、网络访问、系统操作），Agent 运行时的权限不应超过其任务所需
- 输入净化：模型生成的参数可能包含注入攻击（如 SQL 注入、命令注入），工具实现必须对输入进行校验和转义
- 人类确认机制：对于高风险操作（删除文件、发送邮件、支付交易），在执行前需要人类显式确认

### 工具的测试与调试

- 工具调用的测试比普通代码更困难，因为输入由模型动态生成，难以预覆盖所有可能的参数组合
- 测试策略：单元测试（验证工具在合法输入下的正确行为）、边界测试（空输入、超长输入、特殊字符）、对抗测试（模型可能生成的异常参数）
- 调试工具链：调用日志（记录每次工具调用的完整参数和返回值）、回放机制（基于日志重现和调试特定调用序列）、模拟工具（返回预设结果，用于隔离测试模型决策）

## 高级工具调用模式

### 工具组合与链式调用

- 单个工具的能力有限，复杂任务需要多个工具组合使用。工具组合的模式：串联（A 的输出是 B 的输入）、分支（根据 A 的结果选择 B 或 C）、聚合（A 和 B 的结果合并给 C）
- 链式调用的编排：谁来决定调用顺序？模型自行决策（灵活但不可控）vs 预定义工作流（可控但僵化）vs 混合模式（框架定义骨架，模型填充细节）
- 组合调用的原子性问题：链式调用中途失败时，是否需要回滚已完成的步骤？这取决于操作的幂等性和可逆性

### 动态工具生成

- 静态工具集的局限：无法覆盖所有可能的操作。动态工具生成允许根据任务需要，在运行时创建新的工具
- 动态工具的典型场景：代码生成工具（模型生成一段代码作为新工具）、API 适配器（根据 API 文档自动生成调用工具）
- 动态工具的风险：生成的代码可能包含安全漏洞、性能问题或错误行为，需要额外的验证和沙箱机制

### 工具学习与工具创造

- 工具学习（Tool Learning）是指模型通过使用工具积累经验，提升后续工具使用的效率和准确率
- 从工具使用到工具创造的演进：模型不仅使用已有工具，还能根据任务需求设计新工具（如编写一个数据处理脚本来完成特定转换）
- 工具创造需要模型具备编程能力和对任务需求的深层理解，是 Agent 能力的高阶体现
