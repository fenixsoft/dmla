# 记忆系统

LLM 的上下文窗口是有限的，而 Agent 执行的任务可能持续数分钟甚至数小时，产生的信息量远超窗口容量。记忆系统（Memory System）是 Agent 在上下文窗口之外存储和检索信息的机制，让 Agent 能够"记住"过去的经验、积累知识、维持长期状态。这篇文章回答的核心问题是：Agent 如何在有限的上下文窗口之外，有效地存储、检索和利用信息？

## 记忆的分类

### 工作记忆

- 工作记忆（Working Memory）是 Agent 当前推理过程中直接可用的信息，对应 LLM 的上下文窗口
- 工作记忆的容量由上下文窗口大小决定，信息进出遵循 FIFO 或优先级淘汰策略
- 工作记忆的管理挑战：哪些信息应该保留在窗口中、哪些应该移出、移出的信息如何摘要压缩
- 工作记忆与人类短期记忆的类比：容量有限、存取快速、内容易失

### 长期记忆

- 长期记忆（Long-term Memory）是持久化存储的信息，不受上下文窗口限制，需要显式的检索操作才能进入工作记忆
- 长期记忆的存储形式：结构化数据（数据库记录）、非结构化数据（文档、对话历史）、向量嵌入（语义检索）
- 长期记忆的写入时机：任务完成后总结经验、对话中提取关键事实、工具调用结果中保存有用信息
- 长期记忆与人类长期记忆的类比：容量大、存取需检索、持久稳定

### 情景记忆与语义记忆

- 情景记忆（Episodic Memory）记录具体的经历和事件，如"上次部署时遇到了端口冲突问题，通过修改配置文件解决了"
- 语义记忆（Semantic Memory）记录抽象的知识和规则，如"Node.js 应用默认监听 3000 端口"
- 两者的区别：情景记忆是"我经历过什么"，语义记忆是"我知道什么"。情景记忆是具体的，语义记忆是抽象的
- Agent 中两者的转化：反复出现的情景记忆可以抽象为语义记忆（从多次部署经验中总结出"端口冲突是常见问题"的规则）

## 记忆的存储

### 向量存储与语义检索

- 向量存储（Vector Store）是将信息编码为高维向量并存储，检索时通过向量相似度找到语义相关的信息
- 向量存储的工作流程：文本 → 嵌入模型 → 向量 → 存入索引 → 查询向量 → 相似度搜索 → 返回相关文本
- 向量存储的优势：语义检索能力（"如何处理错误"可以匹配到"异常捕获的最佳实践"）、无需精确关键词匹配
- 向量存储的局限：嵌入质量依赖模型能力、相似度不等于相关性（语义相近但内容无关的文档可能被误检索）、索引构建和维护有计算成本

### 结构化存储与精确查询

- 结构化存储（如关系数据库、键值存储）适合存储有明确 schema 的信息，支持精确的条件查询
- 适合结构化存储的信息类型：用户偏好（键值对）、任务状态（状态机记录）、工具调用历史（时间序列）
- 结构化存储的优势：查询精确、事务支持、数据一致性强
- 结构化存储的局限：需要预定义 schema、无法处理模糊查询、语义检索能力弱

### 混合存储架构

- 实际 Agent 系统通常需要向量存储与结构化存储的混合：向量存储负责语义检索，结构化存储负责精确查询
- 混合架构的设计：统一接口层屏蔽底层存储差异，根据查询类型自动路由到合适的存储后端
- 混合检索策略：先向量检索获取语义相关的候选集，再用结构化条件过滤，结合两者的优势
- 数据同步问题：同一信息可能同时存在于向量存储和结构化存储中，需要保证一致性

## 记忆的检索

### 检索时机与触发

- 记忆检索不是被动等待查询，而是主动在合适的时机将相关信息注入 Agent 的上下文
- 检索触发时机：任务开始时（加载相关背景知识）、执行中遇到困难时（搜索类似问题的解决经验）、关键决策前（检索相关历史决策）
- 主动检索 vs 被动检索：主动检索由 Agent 自主判断何时需要信息，被动检索由系统规则触发
- 检索频率的权衡：过于频繁的检索增加延迟和成本，检索不足导致 Agent 缺乏必要信息

### 相关性排序与过滤

- 检索到的信息不一定都有用，需要根据与当前任务的相关性进行排序和过滤
- 排序维度：语义相关性（与当前问题的语义匹配度）、时效性（信息的时效，过时信息应降权）、可靠性（信息来源的可信度）
- 过滤策略：硬过滤（低于阈值的直接丢弃）、软过滤（低相关性的信息压缩后保留摘要）、上下文感知过滤（根据当前对话主题动态调整过滤标准）
- 检索结果的数量控制：返回太多信息会淹没关键内容，返回太少可能遗漏重要信息。通常控制在 3-10 条

### 记忆压缩与摘要

- 长期记忆的原始数据可能非常冗长（如完整的对话历史），直接检索和注入上下文效率低下
- 记忆压缩的策略：摘要（将长文本压缩为关键要点）、提取（只保留关键实体和关系）、聚类（将相似记忆合并为一条代表性记录）
- 压缩的时机：写入时压缩（存储时就生成摘要，节省空间但可能丢失细节）、检索时压缩（检索到原始数据后动态压缩，保留灵活性）
- 压缩的损失：任何压缩都会丢失信息，关键是保留与未来任务相关的信息。这需要预测哪些信息将来有用，而这是不确定的

## 记忆的更新与遗忘

### 记忆的时效性

- 信息有保质期：API 文档可能随版本更新而失效，市场数据可能随时间变化而失真，用户偏好可能随使用习惯而改变
- 时效性管理策略：时间戳标注（每条记忆记录创建和最后验证时间）、定期验证（对长期未验证的记忆重新确认）、自动过期（超过一定时间未使用的记忆降权或删除）
- 时效性与检索排序的关系：在排序时将时效性作为因子，近期验证的信息权重更高

### 冲突检测与解决

- 新信息可能与已有记忆冲突：用户今天说"我喜欢深色主题"，但之前记录的是"用户偏好浅色主题"
- 冲突检测：在写入新记忆时，检索相关已有记忆，比较是否矛盾
- 冲突解决策略：时间优先（新信息覆盖旧信息）、来源优先（高可信度来源覆盖低可信度来源）、合并（保留两者，标注冲突供后续判断）
- 冲突解决是记忆系统中最难的问题之一，因为"正确"本身可能是上下文相关的

### 遗忘机制

- 遗忘（Forgetting）不是记忆系统的缺陷，而是必要的功能。无限积累的记忆会导致检索效率下降、存储成本增加、过时信息干扰决策
- 遗忘策略：基于时间的遗忘（长期未访问的记忆逐渐降权）、基于重要性的遗忘（低重要性记忆优先淘汰）、基于冗余的遗忘（与其他记忆高度重复的优先淘汰）
- 艾宾浩斯遗忘曲线的启发：人类记忆的遗忘速度先快后慢，Agent 记忆可以模拟类似的衰减曲线
- 遗忘的可逆性：软遗忘（降权但不删除，需要时可以恢复）vs 硬遗忘（彻底删除，不可恢复）

## 记忆与上下文的协同

### 上下文窗口管理

- 上下文窗口是 Agent 最宝贵的资源，需要精心管理哪些信息占据窗口空间
- 窗口管理策略：优先级排序（高优先级信息保留在窗口中）、滑动窗口（保留最近 N 轮对话，更早的对话移入长期记忆）、摘要替换（将早期对话压缩为摘要，释放窗口空间）
- 系统提示词（System Prompt）占据固定的窗口空间，需要控制其长度，将可变信息移到动态注入的部分
- 窗口溢出的处理：当信息量超过窗口容量时，需要有明确的淘汰策略，而非随机截断

### 记忆注入的格式

- 从长期记忆检索到的信息需要以合适的格式注入上下文，让模型能有效利用
- 注入位置的选择：系统提示词中（全局背景，始终可见）、用户消息前（任务相关背景）、工具调用结果后（补充上下文）
- 注入格式的最佳实践：结构化标记（用 XML 标签或 Markdown 分隔记忆内容与对话内容）、来源标注（标明信息来自记忆而非当前对话）、时效标注（标明信息的时间属性）
- 注入量控制：注入过多信息会稀释当前对话的信号，注入过少则无法提供足够的背景

### 反思与记忆整合

- 反思（Reflection）是 Agent 主动回顾和整合记忆的过程，将零散的经验提炼为结构化的知识
- 反思的触发时机：任务完成后（总结经验教训）、连续失败后（分析失败模式）、定期（如每完成 N 个任务后进行一次反思）
- 反思的产出：经验规则（"当遇到 X 类问题时，应该先尝试 Y"）、知识更新（修正已有记忆中的错误）、策略调整（优化工具选择或任务分解策略）
- 反思与人类"复盘"的类比：不是简单地记录发生了什么，而是提炼出可复用的规律
