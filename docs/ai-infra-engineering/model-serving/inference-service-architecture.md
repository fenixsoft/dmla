# 推理服务架构原理

## 核心问题

大语言模型的推理服务与传统 Web 服务有何本质不同？为什么一个能在 100ms 内响应的 HTTP API，换成 LLM 推理就可能需要数十秒？理解推理服务的架构设计原理，是构建可靠、高效 AI 应用的基础。本文从 LLM 推理的自回归本质出发，分析推理服务与传统 Web 服务在延迟模型、资源消耗、流量模式上的根本差异，进而讨论推理服务的核心架构组件、部署模式以及高可用设计。

## 第一章：推理服务与传统 Web 服务的差异

### 1.1 延迟模型的本质差异

- 传统 Web 服务的请求延迟基本可预测：数据库查询 + 业务逻辑 + 网络传输，通常在毫秒到秒级，且波动范围有限
- LLM 推理的延迟由生成长度决定，而输出 token 数不可预知。一个"写一首诗"的请求可能生成 200 个 token，一个"详细解释量子力学"的请求可能生成 2000 个 token，延迟相差 10 倍
- 传统服务的延迟呈正态分布（大部分请求集中在均值附近），LLM 推理的延迟呈长尾分布（少数超长请求拖高 P99），这对超时控制、容量规划提出全新挑战

### 1.2 资源消耗模式差异

- 传统 Web 服务是 I/O 密集型，CPU 大部分时间在等网络或磁盘，增加并发几乎不增加 CPU 负担
- LLM 推理是计算密集型 + 显存密集型。每生成一个 token 都需要完整的前向传播，且 KV Cache 占据大量显存，并发数受限于显存容量而非 CPU 核数
- 传统服务可以轻松水平扩展（加机器就能加并发），LLM 推理服务的扩展受 GPU 供应约束，一块 A100 80GB 同时能处理的 70B 模型请求数可能只有个位数

### 1.3 流量模式差异

- 传统 Web 服务的流量波动相对平缓，高峰期与低谷期的比值通常在 2-5 倍
- LLM 推理服务的流量具有突发性。一个热门 AI 应用的日间流量可能是夜间的 10-50 倍，且单个用户的请求间隔极不均匀（可能连续对话，也可能间隔数小时）
- 流量的突发性 + GPU 扩展的缓慢（启动一个 GPU 实例需要数分钟），使得弹性伸缩比传统服务困难得多

## 第二章：推理请求的生命周期

### 2.1 请求处理全流程

- 从客户端发起请求到收到完整回复，一个 LLM 推理请求经过的完整链路：API 网关 → 请求预处理（分词 + 构建 Prompt）→ 调度（选择 GPU 实例）→ Prefill（处理输入）→ Decode（逐 token 生成）→ 流式输出 → 请求完成
- 每个环节的耗时特征和可能的瓶颈点。API 网关的延迟通常在 1-5ms，分词约 10-50ms，调度约 1-10ms，Prefill 取决于输入长度，Decode 取决于输出长度

### 2.2 流式输出与 Server-Sent Events

- LLM 推理的生成是逐 token 进行的，如果等全部生成完毕再返回，用户需要面对漫长的空白等待。流式输出（Streaming）让每生成一个 token 就立即发送给客户端
- SSE（Server-Sent Events）是实现流式输出的标准协议。对比 WebSocket，SSE 是单向推送、自动重连、基于 HTTP，更适合推理结果的流式传输
- 流式输出带来的工程挑战：中间件超时设置（Nginx 默认 60 秒超时）、断线重连与续传、token 级别的错误处理
- 框架实例：TGI（Text Generation Inference）的流式输出实现，通过逐 token 的 SSE 事件将生成结果推送给客户端

### 2.3 请求取消与超时处理

- 用户中断请求（如点击"停止生成"）时，服务端需要立即释放 GPU 资源。传统服务取消请求只需关闭连接，LLM 推理服务还需要回收已分配的 KV Cache 显存
- 超时策略：全局超时（限制最大生成时间）vs 逐步超时（限制单个 token 的最大等待时间）。全局超时可能导致长请求被误杀，逐步超时更精细但实现更复杂
- 请求取消的竞态条件：用户取消与模型生成同时发生时，如何保证 KV Cache 被正确回收而不泄漏

```python runnable
# 演示 SSE 流式输出的模拟（非真实推理，用 sleep 模拟生成过程）
import time
import json

def simulate_streaming_generation(prompt, max_tokens=20):
    """模拟 LLM 的流式生成过程"""
    tokens = ["在", "大", "语", "言", "模", "型", "的", "推", "理", "服",
              "务", "中", "，", "流", "式", "输", "出", "至", "关", "重"]

    events = []
    for i, token in enumerate(tokens[:max_tokens]):
        # 模拟每个 token 的生成延迟
        time.sleep(0.05)
        event = {
            "token": token,
            "token_id": i,
            "timestamp": time.time(),
            "finished": i == len(tokens[:max_tokens]) - 1
        }
        events.append(event)
        # SSE 格式：data: {json}\n\n
        print(f"data: {json.dumps(event, ensure_ascii=False)}\n")

    # 发送结束事件
    print(f"data: {json.dumps({'finished': True, 'total_tokens': len(tokens[:max_tokens])}, ensure_ascii=False)}\n")

simulate_streaming_generation("请介绍LLM推理服务")
```

## 第三章：推理服务的核心架构组件

### 3.1 API 网关层

- API 网关在推理服务中承担的职责：认证鉴权（API Key 验证）、限流（令牌桶/漏桶算法控制 QPS）、请求路由（将请求分发到不同的模型服务实例）
- 限流策略的特殊性：传统 Web 限流按 QPS，LLM 推理还需要按 token 吞吐量限流（TPM，Tokens Per Minute）。一个生成 2000 token 的请求对系统的压力远大于生成 50 token 的请求
- 请求优先级：付费用户 vs 免费用户的差异化调度，通过优先级队列实现

### 3.2 请求调度器

- 调度器是推理服务的"大脑"，负责决定每个请求由哪个 GPU 实例处理
- 调度决策的输入信息：各实例的当前负载（正在处理的请求数、KV Cache 占用量、GPU 利用率）、请求属性（输入长度、预期输出长度、优先级）
- 调度目标：最小化请求延迟、最大化吞吐量、保证公平性。这三个目标常常互相矛盾
- 框架实例：vLLM 的调度策略（FCFS + 抢占机制），当显存不足时暂停低优先级请求以释放资源

### 3.3 模型执行引擎

- 模型执行引擎负责在 GPU 上实际运行推理计算，核心能力包括：模型加载与权重管理、KV Cache 管理、批处理调度
- 执行引擎与调度器的协作关系：调度器决定"哪些请求一起处理"，执行引擎决定"怎么高效地并行计算"
- 框架实例：vLLM 的 PagedAttention 引擎、TensorRT-LLM 的 Kernel 优化引擎

### 3.4 输出处理层

- 输出处理层负责将原始的 token 序列转换为用户可读的文本，包括：分词器解码（token ID → 文本）、特殊标记过滤（移除 `<|im_end|>` 等控制标记）、流式分段（按句子或段落切分输出）
- 输出过滤与安全：内容安全检测、敏感词过滤，在流式输出中逐 token 检测的工程挑战

## 第四章：部署模式与架构选型

### 4.1 单机单卡部署

- 最简单的部署方式：一块 GPU 运行一个模型实例。适合模型较小（7B 以下）且并发需求低的场景
- 局限性：单块 GPU 的显存限制了可部署的模型大小和并发数，没有容错能力，GPU 故障直接导致服务不可用
- 适用场景：开发测试、小规模内部工具

### 4.2 单机多卡部署

- 当模型参数量超过单卡显存时，需要使用张量并行（Tensor Parallelism）将模型分片到多块 GPU 上
- 张量并行的通信开销：每个 Transformer 层的前向传播都需要 AllReduce 操作，GPU 之间的通信带宽成为瓶颈。NVLink（300 GB/s）远优于 PCIe（64 GB/s），因此张量并行通常要求 GPU 在同一节点内通过 NVLink 互连
- 框架实例：vLLM 的张量并行实现，通过 NCCL 完成多卡之间的 AllReduce 通信

### 4.3 多机多卡部署

- 当模型规模超过单机 GPU 总显存（如 70B 模型需要 4×A100 80GB），需要流水线并行（Pipeline Parallelism）将模型的不同层分布到不同机器上
- 流水线并行的气泡问题：各阶段之间需要等待前一个阶段的输出，GPU 存在空闲等待时间。气泡比例与并行度正相关
- PD 分离部署模式：将 [Prefill 和 Decode 分配到不同实例](../../language-models/reasoning/inference-efficiency.md#prefill-decode-分离架构)的工程实践

### 4.4 云原生部署与弹性伸缩

- Kubernetes 上的 LLM 推理服务部署：GPU 节点池管理、模型权重持久化（PVC 或对象存储）、Pod 调度策略
- 弹性伸缩的挑战：GPU 节点启动慢（加载模型权重需要数十秒到数分钟）、缩容时如何优雅处理正在进行的请求
- Serverless GPU 的探索：Cold Start 问题（模型加载延迟）、权重共享机制（多实例复用同一份权重内存）

## 第五章：高可用与容错设计

### 5.1 故障模式分析

- GPU 推理服务的常见故障：GPU 硬件故障（ECC 错误、显存损坏）、OOM（显存溢出）、推理超时、模型权重损坏
- 与传统服务故障的关键区别：GPU 故障通常需要重启整个进程（甚至重新加载模型），无法像 CPU 服务那样简单重启工作线程

### 5.2 冗余与故障转移

- 多副本部署：同一模型部署多个实例，通过负载均衡分发请求。副本数的计算：基于 SLO（如 P99 延迟 < 2s）和峰值 QPS 推算所需的最小 GPU 数量
- 故障检测与自动摘除：健康检查接口（/health）、推理探针（定期发送测试请求验证推理正确性）、自动摘除故障实例
- 请求重试策略：幂等性保证（LLM 推理天然幂等——相同输入产生相同输出分布）、重试风暴防护（指数退避 + 抖动）

### 5.3 降级策略

- 模型降级：当大模型实例不可用时，自动切换到小模型实例（如从 70B 降级到 7B），以牺牲质量换取可用性
- 功能降级：关闭流式输出、降低最大生成长度、拒绝低优先级请求
- 降级触发条件：GPU 利用率持续超过阈值、请求排队时间超过阈值、OOM 频率超过阈值

## 本章小结

- LLM 推理服务与传统 Web 服务在延迟模型、资源消耗、流量模式上存在根本差异，不能简单套用 Web 服务的架构设计
- 推理请求的生命周期包含分词、调度、Prefill、Decode、流式输出等环节，每个环节都可能成为瓶颈
- 推理服务的核心组件是 API 网关、调度器、执行引擎和输出处理层，调度器是整个系统的"大脑"
- 部署模式从单机单卡到云原生弹性，选择依据是模型规模、并发需求和成本预算
- 高可用设计需要考虑 GPU 故障的特殊性，冗余、故障转移和降级策略缺一不可

## 练习题

1. 假设你负责部署一个 7B 参数的 LLM 推理服务，预期峰值 QPS 为 10，每个请求平均生成 500 个 token。请估算需要多少块 A100 80GB GPU（假设单块 A100 运行该模型的 TPS 为 30），并设计部署架构。

   <details>
   <summary>参考答案</summary>

   单块 A100 的 TPS 为 30，10 QPS × 500 token = 5000 token/s 的吞吐需求。理论上需要 5000 / 30 ≈ 167 块 GPU。但这个计算忽略了批处理带来的吞吐提升。实际中，vLLM 等框架通过 PagedAttention 和连续批处理，单块 A100 运行 7B 模型的吞吐量可达 2000-3000 token/s（批量大小约 50-100），因此实际需要约 2-3 块 A100。考虑高可用（至少 2 副本），建议 4-6 块 A100。

   </details>

2. 分析流式输出对推理服务各个组件的影响：API 网关、调度器、执行引擎分别需要做哪些适配？

   <details>
   <summary>参考答案</summary>

   API 网关需要支持 SSE 协议的长连接，调整超时设置（从秒级扩展到分钟级），避免缓冲响应数据。调度器需要感知长连接的存在，请求在 Decode 期间持续占用 GPU 资源，调度器在评估实例负载时应考虑正在流式输出的请求数。执行引擎需要支持 token 级别的输出回调，每生成一个 token 就通知上层，而不是等全部生成完毕。

   </details>

3. 某推理服务在高峰期出现大量请求超时，日志显示 GPU 利用率仅 30%。请分析可能的原因并提出解决方案。

   <details>
   <summary>参考答案</summary>

   GPU 利用率低而请求超时，说明 GPU 没有被充分利用，瓶颈在调度或数据流而非计算本身。可能的原因包括：(1) 调度策略不合理，请求未能有效批处理，大量请求以批量大小 1 执行；(2) Prefill 请求过大，阻塞了 Decode 队列；(3) KV Cache 碎片化严重，显存利用率低，无法接纳更多请求进入批处理。解决方案：(1) 使用连续批处理（如 vLLM 的 iteration-level scheduling），动态合并请求；(2) 限制单个 Prefill 请求的最大长度，或采用 PD 分离架构；(3) 使用 PagedAttention 管理 KV Cache，消除碎片。

   </details>
