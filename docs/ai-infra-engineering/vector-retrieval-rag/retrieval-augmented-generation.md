# 检索增强生成

前面的章节分别讲解了向量检索、混合检索与重排序、文档处理流水线——这些是 RAG 系统的"基础设施"。本文将它们串联起来，讲解如何将外部知识检索与语言模型生成相结合，使模型能够回答训练数据之外的问题、减少幻觉、支持引用溯源。从最基础的检索-注入-生成流程出发，逐步扩展到查询改写、迭代检索、Self-RAG 等高级模式，覆盖从简单到复杂的完整 RAG 设计谱系。

## LLM 的知识边界与 RAG 的动机

LLM 的知识来源于训练数据，有三个根本局限：知识有截止日期，训练后无法获取新信息；对不知道的问题倾向于编造看似合理的答案（幻觉）；专业领域的长尾知识在训练数据中稀少。RAG 的核心思想是让模型"开卷考试"——先检索相关资料，再基于资料回答问题。这与人类面对专业问题时的行为一致：查阅文献后撰写综述，而非仅凭记忆作答。

## 基础 RAG 流程

### 检索-注入-生成三阶段

```
用户查询 → 查询编码 → 向量检索 → 上下文拼接 → LLM 生成 → 回答
```

- **检索**（Retrieval）：将用户查询编码为向量，在索引中检索 Top-k 相关文档
- **注入**（Augmentation）：将检索到的文档拼接到提示词中
- **生成**（Generation）：LLM 基于增强后的提示词生成回答

### 提示词模板

```
基于以下参考资料回答问题。如果无法从资料中找到答案，请说明"根据现有资料无法回答"。

参考资料：
{context}

问题：{query}
回答：
```

模板设计原则：明确指示使用检索内容、处理无法回答的情况、控制输出格式。

### 基础 RAG 的局限

基础 RAG 的三阶段流程简单直接，但在复杂场景中力不从心：单次检索可能遗漏关键信息；检索结果过多会超出 LLM 的上下文长度；多个文档可能包含矛盾信息；模型可能只是拼接检索结果而非真正理解。

## 查询改写与扩展

用户查询可能模糊、不完整或与文档表达方式不匹配——用户用口语提问，文档用专业术语。查询改写与扩展的目标是弥合这一语义鸿沟。

### 查询扩展（Multi-Query）

让 LLM 生成多个相关查询，分别检索后合并结果。从不同角度检索，提高召回率。例如查询"RAG 技术"，扩展为"RAG 技术的原理"、"RAG 技术的应用场景"、"RAG 与微调的对比"，分别检索后合并去重。

### HyDE（Hypothetical Document Embedding）

让 LLM 生成一个假设性答案文档，用该文档的嵌入进行检索。理论基础：假设性答案与真实答案在向量空间中更接近。优势是弥合查询与文档之间的语义鸿沟，劣势是假设性答案可能偏离，引入噪声。

### 查询压缩

将长查询压缩为关键信息，去除冗余，提取核心意图。适用于用户输入冗长或包含无关细节的场景。

### 自查询检索

"2024 年发布的关于 Transformer 的论文"——这类查询同时包含语义条件（"Transformer 论文"）和结构化条件（"2024 年发布"）。纯向量检索无法处理年份过滤，纯标量过滤无法处理语义匹配。自查询检索用 LLM 将查询分解为语义部分（用于向量检索）和结构化部分（用于标量过滤），组合执行。

## 上下文注入策略

### 上下文窗口管理

LLM 的上下文窗口有限（4K-128K token），检索结果的总长度需控制在窗口限制内。策略：按相关性排序，截断到窗口限制内。

### 上下文组织方式

- **简单拼接**：将所有检索结果按顺序拼接
- **结构化组织**：按主题分组，添加标题和分隔符
- **压缩摘要**：对长文档先做摘要再注入
- **对比标注**：对矛盾信息标注来源，让模型判断

### Lost in the Middle

研究表明 LLM 对上下文中间位置的信息关注度较低。信息排列策略：将最相关的信息放在开头和结尾，次相关的放在中间。

### 上下文压缩

长上下文导致延迟增加、成本增加、注意力稀释。压缩方法：提取关键句、LLM 摘要、冗余去除。压缩与信息保留需权衡——过度压缩会丢失细节，压缩不足浪费 Token。

## 迭代检索

复杂问题无法通过单次检索获取全部相关信息。例如"比较 Transformer 和 RNN 在长序列建模上的优劣"，需要分别检索两者的信息，还需检索评估方法，单次检索可能只返回其中一部分。

### 链式检索

第 1 轮检索"Transformer 长序列建模"，第 2 轮基于第 1 轮结果检索"RNN 长序列建模"，第 3 轮检索"两者对比评估方法"。每轮检索基于前轮结果，逐步逼近完整信息。

### 子问题分解

将复杂问题分解为多个子问题，对每个子问题独立检索，合并所有子问题的检索结果。适合"比较 X 和 Y"这类需要多方面信息的查询。

### 迭代精化

生成初步回答 → 检查完整性 → 如果不完整，生成补充查询继续检索 → 重复直到回答完整或达到迭代上限。

### 迭代终止条件

- 回答完整性检查：是否涵盖了问题的所有方面
- 信息收敛检测：新检索结果是否提供了增量信息
- 最大迭代次数：防止无限循环
- 延迟预算：总时间不超过阈值

## Self-RAG

基础 RAG 总是检索、总是使用检索结果。Self-RAG 让模型自主决定：

- 是否需要检索？（有些问题模型本身就能回答）
- 检索结果是否相关？（不相关则不使用）
- 生成结果是否忠实？（不忠实则重新检索）

### 反思标记

Self-RAG 引入四个反思标记，作为模型的自省机制：

- $\texttt{[Retrieve]}$：是否需要检索？$\in \{\texttt{Yes}, \texttt{No}\}$
- $\texttt{[ISREL]}$：检索结果是否相关？$\in \{\texttt{Relevant}, \texttt{Irrelevant}\}$
- $\texttt{[ISSUP]}$：生成是否由检索结果支持？$\in \{\texttt{Supported}, \texttt{Not\_supported}\}$
- $\texttt{[ISUSE]}$：生成是否有用？$\in \{\texttt{Useful}, \texttt{Not\_useful}\}$

```
查询 → 模型判断 [Retrieve]
  ├── No → 直接生成 → 输出
  └── Yes → 检索 → 模型判断 [ISREL]
       ├── Irrelevant → 放弃检索结果，直接生成
       └── Relevant → 生成 → 模型判断 [ISSUP]
            ├── Not_supported → 重新检索或调整生成
            └── Supported → 模型判断 [ISUSE]
                 ├── Not_useful → 调整生成
                 └── Useful → 输出
```

Self-RAG 的训练方式是将反思标记作为特殊 token 加入词表，训练模型学习何时输出这些标记。推理时通过标记控制检索行为，实现检索与生成的自适应协调。

## 多路召回与融合

不同类型的信息分布在不同知识源中：向量索引擅长语义检索，关键词索引擅长精确匹配，知识图谱擅长结构化关系查询，SQL 数据库擅长数值统计。

### 查询路由

根据查询类型选择合适的知识源：事实型查询路由到知识图谱，语义型查询路由到向量索引，精确匹配查询路由到关键词索引，统计型查询路由到 SQL 数据库。路由决策可由 LLM 完成——让 LLM 判断查询类型并选择检索路径。

### 图 RAG

知识图谱提供结构化关系和多跳推理能力，与向量检索的语义泛化能力互补。图 RAG 的流程：从查询中提取实体 → 在知识图谱中检索相关子图 → 沿关系路径多跳推理 → 同时向量检索获取补充信息 → 融合结构化和文本信息生成回答。

## 引用溯源与可信度

RAG 的价值不仅在于回答问题，更在于让回答可验证。无引用的回答用户无法判断信息是否可靠，有引用的回答用户可以追溯来源、验证准确性。

### 引用实现

- **内联引用**：回答中标注 [1]、[2] 等编号
- **段落级引用**：每个段落标注来源
- **句子级引用**：每个句子标注来源（粒度最细）

### 可信度评估

- 来源可信度：权威来源 > 非权威来源
- 一致性检查：多个独立来源是否一致
- 时效性判断：信息是否过时

当检索结果不足时，模型应主动表达不确定性——"根据现有资料，这个问题可能有以下答案..."，避免为不确定的信息给出确定性回答。

## RAG 效果评估

### 评估维度

- **忠实度**（Faithfulness）：生成内容是否与检索文档一致，幻觉率是多少
- **相关性**（Answer Relevance）：生成内容是否回答了用户的问题
- **完整性**（Completeness）：生成内容是否涵盖了问题的所有方面

### 评估方法

人工评估最准确但成本高。LLM-as-Judge 用 GPT-4 等模型自动评估，RAGAS 框架提供自动化评估工具。自动评估与人工评估的一致性需要验证。

### 检索与生成的联合优化

检索质量是生成质量的上限，但检索质量不等于生成质量——好的检索加上差的提示词仍然产生差的回答。优化循环：评估 → 诊断 → 优化 → 再评估。诊断时需区分问题来源：是检索不足（召回率低）还是注入不当（上下文组织差）还是生成缺陷（提示词设计差）。

## 本章小结

RAG 通过检索外部知识增强 LLM，解决知识边界和幻觉问题。从基础的检索-注入-生成三阶段，到查询改写、迭代检索、Self-RAG 等高级模式，RAG 的设计谱系覆盖了从简单到复杂的各类场景。引用溯源让回答可验证，是 RAG 区别于纯生成的关键价值。评估驱动优化——忠实度、相关性和完整性是三个核心维度，诊断时需区分检索、注入和生成各自的问题。

3. 实现迭代检索，对复杂问题分解子问题并逐步检索，对比与单次检索的回答完整性。
4. 实现 Self-RAG 的反思标记决策逻辑，对比与基础 RAG 在不同类型查询上的效果差异。
5. 设计引用溯源机制，验证生成内容是否与检索文档一致，计算幻觉率。
