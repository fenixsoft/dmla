# 数据版本管理

机器学习系统的输出由代码和数据共同决定。代码有 Git 追踪每一次变更，数据却往往只有"最新版"这一个状态——一旦数据被覆盖或损坏，既无法回溯到产生某个模型训练结果的数据快照，也无法对比两次实验之间的数据差异。数据版本管理要解决的就是这个问题：让数据像代码一样可追溯、可对比、可复现。

## 数据版本管理的核心问题

### 代码版本管理 vs 数据版本管理

- 代码版本管理（Git）处理的是小体积、行级变更的文本文件，增量存储成本极低
- 数据版本管理面对的是 GB~TB 级的二进制文件、表格数据、图像目录，传统 diff 几乎无法工作
- 数据变更的粒度与代码不同：代码关注行级 diff，数据关注记录级（新增/删除/修改了哪些行）、文件级（新增/删除了哪些文件）和语义级（特征分布是否发生了变化）
- 二者的共性：都需要版本历史、分支、合并、标签等基本能力

### 数据谱系：从原始数据到模型

- 数据在 ML 系统中经历采集、清洗、特征工程、训练集/验证集/测试集划分等多个阶段，每个阶段的输出都是下一阶段的输入
- 数据谱系（Data Lineage）记录数据从源头到消费端的完整流转路径
- 没有版本管理，谱系链条上的任何一环断裂，都无法复现下游结果
- 数据版本管理是构建数据谱系的基础设施：为谱系中的每个节点赋予可引用的版本标识

### 不可变数据与可变数据的差异

- 原始数据（Raw Data）应视为不可变资产，任何清洗、变换后的数据都应生成新版本而非覆盖
- 特征数据、标签数据属于可变数据，每次变更需要记录变更原因和影响范围
- 不可变数据适合内容寻址存储（类似 Git 的 blob 存储），可变数据需要快照与增量结合
- 区分不可变与可变，决定了存储策略和版本粒度的设计

## 数据快照与增量追踪

### 全量快照策略

- 每次数据变更时保存一份完整的副本，实现简单，恢复速度快
- 存储成本随版本数线性增长，对大规模数据集不可持续
- 适用于数据量小（<1GB）或变更频率低的场景
- 全量快照的存储优化：使用列式存储格式（Parquet、ORC）配合压缩，可减少单副本体积

### 增量追踪策略

- 仅记录每次变更的差异部分（新增记录、删除记录、修改前后值），存储成本与变更量成正比
- 增量追踪的实现方式：Delta Log（操作日志）、Change Data Capture（CDC）、时间旅行查询
- 增量恢复需要回放从基准版本到目标版本之间的所有变更，恢复时间与版本跨度成正比
- 基准版本（Checkpoint）的设置策略：每 N 个增量版本创建一次全量快照，平衡存储成本与恢复速度

### 内容寻址存储

- 借鉴 Git 的对象存储模型：将数据文件按内容哈希（SHA-256）命名存储，相同内容只存一份
- 目录结构通过 Tree 对象组织，Tree 记录文件名到内容哈希的映射
- 版本间的 diff 转化为 Tree 对象的比较，可以快速定位变更的文件
- 大文件的内容寻址：将文件分块（Chunk），按块哈希去重，避免整文件级别的重复存储
- 去重效率分析：数据集版本迭代时，90% 以上的数据块通常未变更，内容寻址可将存储开销降低一个数量级

## 数据分支与合并

### 数据分支的场景

- 特征实验分支：在主干数据集上派生出分支，尝试新的特征工程方案，实验失败则丢弃分支
- 数据修复分支：发现历史数据存在标注错误，创建修复分支进行批量修正，验证后合并回主干
- 多租户分支：不同业务线使用同一基础数据集，各自维护业务特定的扩展字段
- 分支的生命周期管理：短期实验分支应定期清理，避免存储膨胀

### 数据合并的挑战

- 代码合并可以逐行比较，数据合并需要处理记录级的冲突：两个分支修改了同一条记录的不同字段
- 三路合并在数据上的扩展：基准版本、分支 A、分支 B 的三方比较，自动合并无冲突的变更，标记有冲突的记录
- 语义冲突：数值修改在语法上无冲突（两个分支修改了不同字段），但在语义上可能矛盾（同一用户的年龄和出生日期不一致）
- 合并策略设计：字段级合并（细粒度，冲突少但复杂）、记录级合并（粗粒度，简单但冲突多）、语义校验合并（合并后自动校验数据一致性）

### 数据标签与发布管理

- 数据标签（Tag）为特定版本赋予语义化的名称，如 v1.0-production、v2.1-experiment-baseline
- 发布管理流程：dev 分支验证 -> staging 环境验证 -> 打标签发布到 production
- 数据不可变性约束：已发布标签对应的数据版本应被锁定，禁止修改，确保可复现性
- 标签与模型版本的关联：每个模型版本应记录其训练数据的标签，构成可追溯的关联关系

## 大规模数据版本管理的工程实践

### 元数据与数据分离

- 元数据（版本号、哈希、变更日志、谱系关系）体积小，存入数据库或版本控制系统
- 数据本体（文件、表格）体积大，存入对象存储（S3、HDFS）或专用数据湖
- 元数据层提供查询、对比、分支等操作接口，数据层提供存储和传输
- 分离的好处：元数据操作无需移动大量数据，版本对比只需比较元数据

### 存储后端的选择

- 对象存储（S3/OSS）：适合大规模文件存储，支持版本控制，但原生不支持增量 diff
- 数据湖格式（Delta Lake、Apache Iceberg、Apache Hudi）：在对象存储之上提供 ACID 事务、时间旅行、增量查询
- 三大数据湖格式的设计差异对比：Iceberg 的快照隔离、Delta Lake 的事务日志、Hudi 的增量拉取
- 选择依据：写入模式（批写入 vs 流写入）、查询模式（全表扫描 vs 增量查询）、生态兼容性

### 版本管理在 ML Pipeline 中的集成

- Pipeline 入口处自动记录输入数据的版本哈希，作为实验元数据的一部分
- Pipeline 出口处自动提交输出数据的版本，关联输入版本和代码版本
- 数据版本作为实验可复现性的关键约束：同一代码 + 同一数据版本 = 可复现结果
- CI/CD 集成：数据版本变更触发下游 Pipeline 的自动验证和重新训练

## 可视化建议

- **数据版本演变图**：时间线视图，展示数据集从 v1 到 vN 的版本演变，标注每次变更的类型（新增数据、修正标签、特征工程）和规模
- **分支与合并图**：类似 Git 分支图的数据分支可视化，展示实验分支的创建、合并和废弃
- **增量 diff 可视化**：两个版本之间的记录级差异展示，高亮新增（绿色）、删除（红色）、修改（黄色）的记录
- **数据谱系图**：DAG 图，展示数据从采集到模型训练的流转路径，节点为数据版本，边为变换操作
