# 模型注册与生命周期

一个模型从实验阶段走向生产环境，要经历验证、审批、部署、监控、退役等多个阶段，就像一件产品从研发到上市再到退市的全生命周期。模型注册表（Model Registry）是这个生命周期的管理中枢：它记录模型的身份信息，管理版本的流转状态，控制从开发到生产的晋升路径。没有模型注册表，模型文件散落在实验服务器的各个目录中，谁也不知道生产环境跑的是哪个版本，出了问题也无法回溯。

## 模型注册的核心问题

### 模型管理的混乱现状

- 模型文件以 `model_v2_final_final.pth` 这样的命名散落在文件系统中，版本语义模糊
- 实验阶段的模型与生产部署的模型之间没有清晰的界限，任何人都可以替换生产模型
- 模型与训练数据的关联缺失：知道模型文件的位置，不知道它是用什么数据训练的
- 模型的部署状态不透明：哪些模型在线上服务？各自服务多少流量？性能如何？

### 模型注册表的定位

- 模型注册表是模型的"户籍管理系统"：每个模型有唯一标识、版本号、状态标签和完整的元数据
- 模型注册表 vs 实验追踪：实验追踪记录每次训练运行（包括失败的），模型注册表只记录通过验证的模型版本
- 模型注册表 vs 模型服务：模型注册表管理模型的存储和元数据，模型服务负责模型的加载和推理
- 三者的协作：实验追踪 -> 模型注册（晋升）-> 模型服务（部署），构成模型的完整流转链路

### 模型唯一标识与版本

- 模型的命名规范：`<domain>/<task>/<architecture>`，如 `recommendation/click-prediction/dcn-v2`
- 语义化版本号：Major（架构变更）. Minor（超参数变更）. Patch（数据重训练）
- 版本与实验的关联：每个模型版本对应一次或多次实验运行，保留训练过程的完整记录
- 模型的指纹：对模型权重计算哈希值，确保模型文件未被篡改

## 模型版本管理

### 版本的创建与继承

- 版本创建的触发条件：训练完成且指标达到阈值、手动从实验追踪系统晋升、定期重训练自动注册
- 版本继承：新版本默认继承上一版本的元数据（特征列表、推理接口），仅覆盖变更部分
- 版本的不可变性：已注册的模型版本不可修改权重和元数据，只能创建新版本
- 版本的元数据：训练数据版本、代码版本、超参数、训练指标、验证指标、注册时间、注册人

### 版本的标注与描述

- 版本标签（Tag）：`baseline`、`champion`、`challenger`、`deprecated` 等语义化标签
- 版本描述：自由文本，记录版本的变更原因、预期改进、已知限制
- 模型签名（Model Signature）：定义模型的输入输出 Schema，包括字段名、类型、形状
- 模型签名的作用：部署时自动校验推理请求是否符合预期格式，防止特征错配

### 版本的搜索与发现

- 按任务搜索：查找所有推荐系统相关的模型
- 按指标排序：按验证集 AUC 降序排列，快速找到最佳版本
- 按状态过滤：查找所有处于 Staging 状态的模型版本
- 按时间范围：查找最近一个月注册的模型版本
- 模型目录（Model Catalog）：团队级别的模型注册中心，支持跨项目发现和复用

## 模型生命周期状态机

### 标准状态流转

- **None** -> **Registered**：训练完成，模型版本首次注册
- **Registered** -> **Staging**：模型通过基本验证，进入预发布环境测试
- **Staging** -> **Production**：模型通过预发布验证，晋升为生产版本
- **Staging** -> **Archived**：预发布验证未通过，归档备查
- **Production** -> **Archived**：被新版本替换，或主动退役
- **Archived** -> **Registered**：归档模型重新激活（少见，通常用于紧急回滚）

### 状态流转的权限控制

- Registered -> Staging：开发人员可以执行，完成基本功能测试即可
- Staging -> Production：需要审批，通常由团队负责人或 ML 平台管理员确认
- Production -> Archived：需要审批，确保线上服务不受影响
- 权限模型：基于角色的访问控制（RBAC），区分开发、测试、运维、管理员等角色

### Champion-Challenger 模式

- Champion：当前线上服务的模型版本
- Challenger：候选的替代模型版本，在 Staging 环境中等待验证
- 多 Challenger 并存：同时测试多个候选版本，选择表现最优的晋升
- 流量分配：Challenger 先接收少量流量（如 1%），逐步增加到 10%、50%、100%
- Champion 的保留策略：新 Champion 上线后，旧 Champion 不立即归档，保留一段时间作为回滚选项

## 模型审批与部署

### 模型验证检查清单

- 功能验证：模型能正常加载和推理，输出格式符合签名定义
- 性能验证：离线指标（AUC、F1）达到预设阈值
- 公平性验证：模型在不同子群体上的性能差异在可接受范围内
- 安全验证：模型对对抗样本的鲁棒性、输出不存在敏感信息泄露
- 兼容性验证：新版本的推理接口与现有服务兼容，特征依赖可满足

### 审批流程设计

- 单节点审批：一个人确认即可晋升，适合小团队
- 多节点审批：多个角色依次确认（开发验证 -> 测试验证 -> 运维审批），适合大团队
- 自动审批：满足预设条件（指标阈值、测试通过）自动晋升，适合高频迭代场景
- 审批记录：每次审批的操作人、时间、审批意见应被完整记录

### 部署策略与模型注册表的协作

- 蓝绿部署：新旧两个版本同时在线，通过流量切换实现平滑过渡，模型注册表维护两个版本的 Active 状态
- 金丝雀发布：新版本先接收少量流量，逐步扩大，模型注册表记录流量分配比例
- 影子部署：新版本接收线上流量但不返回结果，仅用于对比验证，注册表中标记为 Shadow 状态
- 部署回滚：注册表记录历史 Production 版本，支持一键回滚到指定版本

## 模型退役与归档

### 退役的触发条件

- 模型性能持续下降，无法通过重训练恢复
- 业务需求变更，模型不再需要
- 合规要求，模型必须下线
- 技术栈迁移，模型格式不再被支持

### 归档与保留策略

- 归档模型从常规查询结果中隐藏，但保留完整记录供审计和回溯
- 模型权重的保留期限：生产模型退役后保留 90 天（支持回滚），之后可移入冷存储
- 元数据的永久保留：即使模型权重已删除，元数据（训练参数、指标、审批记录）应永久保留
- 存储成本与合规要求的平衡：法律或行业规定可能要求保留模型权重一定期限

### 模型血缘与影响分析

- 模型血缘：记录模型的训练数据来源、上游模型（如预训练模型、蒸馏教师模型）
- 影响分析：当上游数据或模型出问题时，快速定位受影响的下游模型和服务
- 血缘图的可视化：DAG 图展示模型间的依赖关系
- 血缘数据的维护：血缘关系应在模型注册时自动提取，而非手动维护
