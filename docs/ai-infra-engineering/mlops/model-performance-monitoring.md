# 模型性能监控

模型一旦部署到生产环境，故事才刚刚开始。训练时 AUC 0.95 的模型，上线后可能悄然降到 0.85——不是因为代码有 bug，而是因为世界在变：用户行为在变，数据分布漂移了，模型在训练时见过的模式正在失效。模型性能监控要做的，就是持续追踪模型在生产环境中的真实表现，在性能退化演变为业务事故之前发出预警。

## 模型性能监控的核心问题

### 训练性能与生产性能的鸿沟

- 离线指标（训练集/测试集上的 AUC、F1）与在线指标（生产环境中的实际效果）之间存在系统性差异
- 差异来源：训练数据与生产数据的分布差异、特征计算的训练-推理偏差、用户行为的时间演变
- 离线指标的欺骗性：测试集上的高分不代表生产环境中的可靠表现，测试集通常是历史数据的快照，不反映未来
- 真正需要关注的是模型在生产环境中的持续表现，而非训练时的一次性得分

### 延迟标签问题

- 分类和回归模型的性能评估需要真实标签，但生产环境中标签往往延迟获取
- 推荐系统：用户是否点击推荐内容需要等待用户行为，可能延迟数分钟到数天
- 信用评分：贷款是否违约需要等待还款周期结束，延迟可达数月
- 延迟标签导致性能评估滞后：模型可能已经退化，但标签尚未到达，无法及时发现问题
- 延迟标签的处理策略：用代理指标替代真实标签、用部分标签做快速评估、用统计方法估计标签分布

### 监控粒度与成本

- 全局监控：所有请求的聚合指标，成本低但可能掩盖局部问题
- 分片监控：按用户群体、地域、时间段等维度分片计算指标，能发现局部退化但成本更高
- 采样 vs 全量：全量计算成本高，采样可能漏掉小比例但高影响的异常
- 监控粒度的选择取决于业务影响和成本预算：高价值场景（如金融风控）需要更细粒度的监控

## 性能指标的监控体系

### 模型级指标

- 准确性指标：准确率、精确率、召回率、F1-Score、AUC 等，需要真实标签才能计算
- 概率校准指标：Brier Score、Expected Calibration Error（ECE），评估模型输出概率的可靠性
- 排序指标：NDCG、MAP 等，评估排序模型的输出质量
- 回归指标：MAE、RMSE、MAPE 等，评估回归模型的预测精度
- 这些指标的共同前提：需要真实标签，在延迟标签场景下无法即时计算

### 代理指标

- 预测置信度分布：模型输出的最大概率（或熵）的统计分布变化，可能暗示输入数据分布漂移
- 预测类别分布：分类模型输出的类别比例变化，可能反映数据漂移或模型退化
- 特征分布统计：输入特征的均值、方差、分位数等统计量的变化
- 代理指标的优势：无需真实标签，可以实时计算；劣势：间接度量，可能产生误报

### 业务级指标

- 模型性能最终服务于业务目标，业务指标的监控比模型指标更直接
- 推荐系统：点击率、转化率、用户停留时长
- 搜索系统：搜索满意度、无结果率、点击位置
- 风控系统：欺诈检出率、误报率、资损金额
- 业务指标与模型指标的关联：模型指标下降不一定立即导致业务指标下降（业务有缓冲），但持续下降最终会传导

## 性能退化检测

### 突变检测

- 性能指标的突然下降通常对应特定事件：数据管道故障、特征存储异常、模型文件损坏
- 突变检测方法：阈值告警（指标低于设定值）、变化率告警（指标下降速度超过阈值）、统计检验（当前窗口与历史窗口的分布差异检验）
- 阈值设定的挑战：阈值太高产生过多误报，阈值太低漏掉真实的退化
- 动态阈值：基于历史数据的统计特性自适应调整阈值，如均值减 3 倍标准差

### 渐进退化检测

- 渐进退化更常见也更隐蔽：数据分布缓慢漂移，模型性能逐渐下降，日环比变化微小但月累计变化显著
- 趋势检测：线性回归拟合性能指标的时间趋势，检验斜率是否显著为负
- 移动窗口对比：当前窗口的均值与更长的历史窗口对比，检测缓慢偏移
- 渐进退化的阈值：日环比 0.1% 的下降在 30 天后累积为 3%，需要在日级别就发出预警

### 分片退化检测

- 全局指标可能掩盖局部退化：整体 AUC 保持 0.90，但某个用户群体的 AUC 已降至 0.70
- 关键分片维度：用户群体（新用户 vs 老用户）、地域、设备类型、时间段
- 分片监控的统计挑战：小分片的样本量少，指标波动大，容易产生误报
- 分片退化的根因分析：定位退化发生的分片后，追溯该分片的数据来源和特征分布变化

## 监控告警与响应

### 告警策略设计

- 告警分级：P0（立即响应，模型完全失效）、P1（1 小时内响应，性能显著下降）、P2（1 天内响应，轻微下降）
- 告警去重与聚合：同一问题触发的多条告警应聚合为一条，避免告警风暴
- 告警抑制：已知维护期间抑制告警，避免无效通知
- 告警升级：P2 告警持续未处理时自动升级为 P1

### 根因分析框架

- 模型性能退化的可能原因：数据漂移、特征管道故障、模型部署错误、标签定义变更、上游服务异常
- 根因分析的系统方法：检查数据质量 -> 检查特征分布 -> 检查模型输出分布 -> 检查部署配置
- 自动化根因分析：将常见退化模式与根因建立映射，部分场景可自动定位
- 根因分析的输出：退化原因、影响范围、修复建议

### 性能退化后的应对

- 紧急回滚：回退到上一个已知正常的模型版本
- 数据修复：如果是数据质量问题导致退化，修复数据后重新训练
- 模型重训练：如果是数据漂移导致退化，用最新数据重新训练模型
- 应急降级：模型性能不足时切换到规则引擎或默认策略，保证服务可用

## 监控系统架构

### 指标采集管道

- 推理日志收集：每次推理请求的输入特征、模型输出、延迟等信息
- 标签回填管道：延迟到达的真实标签与推理日志的关联匹配
- 指标聚合：实时流式聚合（分钟级）和批量聚合（小时/天级）
- 采样策略：全量采集（高价值场景）vs 采样采集（成本控制）

### 指标存储与查询

- 时序数据库存储指标数据，支持高效的时间范围查询和聚合计算
- 热数据与冷数据分离：最近 7 天的数据存热存储（SSD），更早的数据存冷存储（对象存储）
- 指标查询的常见模式：时间范围查询、分片对比、同比环比计算
- 查询性能优化：预聚合（按小时/天预计算常用指标）、物化视图、索引优化

### 监控仪表盘设计

- 总览面板：全局健康状态、关键指标趋势、活跃告警数量
- 模型详情面板：单个模型的性能指标时间线、分片指标对比、近期变更记录
- 告警管理面板：活跃告警列表、历史告警统计、告警响应时间
- 根因分析面板：关联数据质量指标、特征分布指标、系统指标（延迟、错误率）
