# 在线实验与渐进发布

离线评估告诉你模型"能"做什么，在线实验告诉你模型"实际"做了什么。一个离线 AUC 提升 0.02 的模型，上线后可能因为延迟增加导致用户体验下降，净效果为负。在线实验通过真实流量验证模型效果，渐进发布通过逐步放量控制风险——二者结合，确保模型改进从"纸面上的提升"变成"用户可感知的改善"。

## 在线实验的核心问题

### 离线评估的局限性

- 离线数据是历史的快照，不包含模型自身对用户行为的反作用（推荐系统改变了用户的点击模式，但离线数据中不存在这种交互）
- 离线指标与业务指标的偏差：AUC 提升 0.01 不等于点击率提升 0.01，二者之间的映射关系复杂
- 位置偏差（Position Bias）：离线数据中排在前面的内容更容易被点击，但这不代表它更相关
- 选择偏差（Selection Bias）：离线数据只包含用户看到并与之交互的内容，模型推荐的新内容没有交互记录

### 在线实验的必要性

- 在线实验是验证模型真实效果的唯一可靠方式：用真实流量、真实用户、真实环境测试
- 因果推断的基础：在线实验通过随机分组实现因果效应的估计，而非相关性
- 发现预期外影响：模型可能在目标指标上提升，但在辅助指标上退化（如推荐更吸引人的标题提升了点击率但降低了内容质量）
- 在线实验的成本：需要工程基础设施支持、需要足够的流量和实验时长、可能影响部分用户体验

### 实验的伦理与合规

- 用户分组的公平性：实验组和对照组的分配应随机且公平，不应基于敏感属性（种族、性别）分组
- 实验风险的评估：实验可能对部分用户造成负面影响（如推荐系统实验可能降低某些用户的信息获取质量）
- 用户知情与同意：某些行业或地区要求用户知晓并同意参与实验
- 实验时长与样本量的平衡：实验越长统计功效越高，但对用户的潜在影响时间也越长

## A/B 测试

### A/B 测试的基本框架

- 随机将用户分为对照组（A 组，使用当前模型）和实验组（B 组，使用新模型）
- 核心假设：在随机分组的前提下，两组用户的差异仅由模型差异导致
- 指标比较：计算两组用户在目标指标上的差异和统计显著性
- A/B 测试的黄金标准：随机化 + 对照 + 足够样本量 = 因果推断

### 样本量与统计功效

- 统计功效（Power）：当真实效应存在时，实验能检测到的概率，通常要求 >= 80%
- 最小可检测效应（MDE）：实验能可靠检测的最小指标差异，由业务需求决定
- 样本量公式：n = (Z_alpha/2 + Z_beta)^2 * 2 * sigma^2 / delta^2，其中 delta 是 MDE，sigma 是指标标准差
- 样本量的实践估算：给定 MDE 和指标方差，计算所需样本量和实验时长
- 流量有限时的策略：延长实验时间、合并相似实验、使用方差缩减技术

### 统计显著性检验

- 假设检验框架：H0（两组无差异）vs H1（两组有差异）
- p 值的含义：在 H0 为真的前提下，观察到当前或更极端结果的概率
- 置信区间：指标差异的区间估计，比 p 值提供更多信息（效应大小和精度）
- 多重比较问题：同时检验多个指标时，假阳性概率膨胀，需要 Bonferroni 校正或 FDR 控制
- 统计显著 vs 业务显著：统计显著只说明差异不是偶然，不说明差异足够大以至于值得上线

### A/B 测试的常见陷阱

- 新奇效应（Novelty Effect）：用户对新功能的好奇心导致短期指标提升，长期回归正常
- 首日效应（First-Day Effect）：实验首日的指标波动通常较大，不应据此做决策
- 泄露效应（Spillover Effect）：实验组用户的行为影响对照组用户（如社交网络中的信息传播），违反独立假设
- 样本比率偏差（Sample Ratio Mismatch）：实际分组比例偏离预期比例，通常由技术 bug 导致
- 偷窥问题（Peeking Problem）：实验未结束时提前查看结果并做出决策，导致假阳性率膨胀

## 高级实验设计

### 多臂老虎机（Multi-Armed Bandit）

- A/B 测试的局限：实验期间固定流量分配，即使某个变体明显更差也继续分配 50% 流量
- MAB 的思路：在探索（尝试新变体）和利用（将流量分配给当前最优变体）之间动态平衡
- epsilon-Greedy：以 epsilon 概率探索，1-epsilon 概率利用当前最优
- UCB（Upper Confidence Bound）：选择均值 + 置信区间上界最大的变体，自动平衡探索与利用
- Thompson Sampling：贝叶斯方法，从后验分布中采样选择变体，理论最优且实现简单
- MAB vs A/B 测试的取舍：MAB 优化实验期间收益（减少机会成本），但统计结论不如 A/B 测试严谨

### 交叉实验与因子设计

- 当需要同时测试多个因素（如模型版本 + UI 布局）时，独立 A/B 测试无法捕捉因素间的交互效应
- 全因子设计：测试所有因素组合，如 2 个模型 x 2 种 UI = 4 个实验组，可以分析主效应和交互效应
- 部分因子设计：只测试部分组合，减少实验组数量但损失部分交互信息
- 交叉实验的流量需求：实验组数量增加，每个组的样本量减少，需要更多流量或更长实验时间

### 分层实验与正交实验

- 大型系统同时运行多个实验时，实验之间可能互相干扰
- 分层实验（Layered Experiment）：将实验分配到不同的层，每层内的实验互斥，跨层的实验正交
- 正交分配：不同层的实验对用户的分组是独立随机的，确保跨层实验的独立性
- Google 的重叠实验基础设施（Overlapping Experiment Infrastructure）：经典的多层实验系统设计
- 分层设计的工程实现：用户 ID 哈希 + 层级盐值，确保同一用户在不同层中的分组独立

## 渐进发布

### 渐进发布的阶段

- 阶段一：内部测试（Dogfood）—— 团队成员使用新模型，验证基本功能
- 阶段二：影子部署（Shadow）—— 新模型接收线上流量但不返回结果，仅对比输出差异
- 阶段三：小流量测试（Canary）—— 新模型接收 1%-5% 的线上流量，返回真实结果，密切监控
- 阶段四：逐步放量（Ramp-up）—— 流量从 5% -> 10% -> 25% -> 50% -> 100%，每阶段观察指标
- 阶段五：全量发布（Full Rollout）—— 新模型替换旧模型，旧模型保留为回滚选项

### 流量分配策略

- 用户级分配：同一用户始终路由到同一模型版本，保证体验一致性
- 请求级分配：每个请求独立分配，简单但同一用户可能看到不同模型的结果
- 分层分配：用户先被分配到实验层，再在层内分配到模型版本，支持并行实验
- 权重动态调整：根据实时监控指标自动调整流量比例，性能差则减少流量

### 回滚机制

- 回滚的触发条件：关键指标显著下降、服务错误率上升、用户投诉激增
- 自动回滚：预设回滚阈值，指标越界时自动切换回旧模型
- 手动回滚：人工判断后触发回滚，适用于指标变化不明确但直觉上不放心的场景
- 回滚的时效性：从触发回滚到生效的延迟应尽可能短（秒级），减少业务影响
- 回滚后的复盘：记录回滚原因和观察到的现象，为后续优化提供输入

### 渐进发布的监控

- 每个放量阶段都应有对应的监控指标和告警阈值
- 放量阶段的指标对比：实验组 vs 对照组的指标差异，关注差异是否随流量比例变化
- 延迟监控：新模型的推理延迟是否在可接受范围内，是否随流量增加而劣化
- 资源监控：新模型的 CPU/GPU/内存占用是否符合预期，是否需要扩容
- 长尾效应监控：放量到 100% 后继续观察至少一个完整周期（如一周），确认无延迟出现的问题

## 实验基础设施

### 实验平台的架构

- 实验配置服务：定义实验的分组规则、流量比例、目标指标
- 流量路由服务：根据实验配置将请求路由到对应的模型版本
- 指标收集服务：收集实验组和对照组的指标数据
- 统计分析服务：计算指标差异、置信区间和 p 值
- 实验管理界面：创建、管理和监控实验的 Web UI

### 实验的元数据管理

- 每个实验应记录：实验名称、目标、假设、分组规则、流量比例、开始时间、结束时间、目标指标、辅助指标
- 实验结果应记录：各组的指标值、差异、置信区间、p 值、决策（上线/放弃/继续）
- 实验历史：所有实验的完整记录，避免重复实验和知识流失
- 实验与模型的关联：实验结果关联到模型注册表中的版本，形成闭环

### 实验文化的建立

- 数据驱动的决策文化：任何模型变更都应通过在线实验验证，而非凭直觉上线
- 实验的规范与流程：实验申请 -> 配置 -> 执行 -> 分析 -> 决策的标准化流程
- 实验结果的分享：团队内部定期分享实验结果和经验教训
- 实验能力的建设：培养团队的实验设计、统计分析和因果推断能力
