# 实验追踪

机器学习开发是高度实验性的过程：调整一个超参数、换一种特征组合、尝试一个新的数据预处理方案，每次尝试都是一次实验。问题在于，几周下来你可能做了上百次实验，却无法回答最基本的问题——"上周三那个 AUC 0.92 的模型是怎么训练出来的？用的什么数据？什么参数？"实验追踪要做的，就是为每一次实验建立完整的档案，让实验过程可追溯、可对比、可复现。

## 实验追踪的核心问题

### 实验管理的混乱现状

- 机器学习实验的典型场景：研究人员在 Notebook 中尝试了几十种参数组合，最终选定的模型参数只存在于那次运行的终端输出中
- "笔记驱动"的实验管理：用 Excel、Notion 或纸质笔记记录实验参数和结果，信息分散且不完整
- 不可复现的实验：代码改了、数据改了、环境改了，过去的结果再也无法复现
- 协作困难：团队成员之间的实验不透明，可能重复做相同的实验，也可能遗漏关键对比

### 实验追踪的完整性要求

- 一次实验的完整记录应包含：代码版本、数据版本、超参数、训练配置（硬件、框架版本）、度量指标、输出产物（模型权重、日志）
- 完整性的最小标准：给定实验记录，应能在相同环境下精确复现该实验的结果
- 常见的遗漏项：随机种子未固定、依赖版本未记录、数据预处理步骤未纳入版本控制
- 隐性变量的追踪：GPU 型号、CUDA 版本、确定性算法开关等看似无关但可能影响结果的变量

### 实验追踪 vs 模型注册

- 实验追踪关注过程：每一次训练运行的参数和结果
- 模型注册关注产物：通过验证的模型版本及其元数据
- 实验追踪是模型注册的上游：优秀的模型从实验中筛选出来，注册到模型仓库
- 二者的衔接：实验追踪系统中的"推广"（Promote）操作，将实验记录关联到模型注册表

## 实验记录的元数据模型

### 参数与配置

- 超参数：学习率、批量大小、训练轮次、优化器类型等，直接影响模型性能的可调参数
- 模型配置：网络架构、层数、隐藏维度、激活函数等模型结构定义
- 数据配置：数据集版本、预处理参数、训练/验证/测试集划分策略
- 环境配置：Python 版本、依赖包版本、GPU 型号、分布式策略
- 参数的层次结构：全局参数 -> 模型参数 -> 训练参数 -> 数据参数，支持继承和覆盖

### 度量指标

- 标量指标：Loss、Accuracy、F1-Score 等，随训练步数变化，需要记录完整的时间序列
- 聚合指标：最佳验证集 Loss、最终测试集 Accuracy 等，用于实验间横向对比
- 自定义指标：业务指标（如推荐系统的点击率、搜索系统的 NDCG）、复合指标
- 指标的记录频率：每步记录（细粒度，存储量大）、每轮记录（平衡）、仅记录最终值（粗粒度）

### 产物与输出

- 模型权重：训练产出的模型参数文件，是实验最重要的输出
- 训练日志：终端输出、异常堆栈、警告信息
- 可视化产物：训练曲线图、特征重要性图、混淆矩阵等
- 产物的存储策略：大文件（模型权重）存对象存储，小文件（日志、图片）可存数据库
- 产物与实验的关联：通过实验 ID 关联，确保产物可追溯

## 实验对比与搜索

### 实验对比维度

- 参数对比：哪些参数不同，哪些参数相同，差异参数对指标的影响
- 指标对比：不同实验的性能指标横向比较，支持多指标排序和帕累托前沿分析
- 训练曲线对比：不同实验的 Loss/Accuracy 曲线叠加，观察收敛速度和稳定性
- 成本对比：训练时间、GPU 小时数、数据量等资源消耗的对比

### 实验搜索与过滤

- 按参数搜索：查找所有学习率 > 0.001 且批量大小 = 32 的实验
- 按指标过滤：查找验证集 AUC > 0.9 的实验
- 按时间范围：最近一周的实验、特定日期之后修改了数据版本的实验
- 按标签：为实验打标签（如 baseline、production、experiment），按标签筛选
- 组合查询：参数 + 指标 + 时间的组合过滤，支持复杂搜索场景

### 实验组与实验族

- 实验组（Experiment Group）：一组相关实验的集合，如"学习率调优组"包含 10 个不同学习率的实验
- 实验族（Experiment Family）：具有继承关系的实验，如从 baseline 逐步改进的实验链条
- 分组的意义：便于对比同一变量（如学习率）对指标的影响，排除其他变量的干扰
- 分组对比的可视化：参数-指标散点图、指标排名表、训练曲线叠加图

## 实验追踪的工程实现

### 自动化记录

- 手动记录实验信息容易遗漏，自动化记录是可靠性的基础
- 代码级集成：在训练脚本中调用追踪 SDK，自动记录参数、指标和产物
- 框架级集成：通过 Callback/Hook 机制，自动捕获框架的训练事件（如 epoch 结束时的指标）
- 环境级集成：自动检测并记录运行环境信息（GPU 型型、依赖版本、代码 Git 哈希）
- 自动化 vs 手动：自动化记录客观信息（参数、指标），手动补充主观信息（实验目的、观察结论）

### 实验追踪的存储架构

- 元数据存储：实验参数、指标、配置等结构化数据，存入关系数据库或文档数据库
- 时序数据存储：训练过程中记录的步级指标，存入时序数据库以支持高效查询
- 产物存储：模型权重、日志等大文件，存入对象存储（S3、MinIO）
- 存储的扩展性：实验数量随项目增长，需支持百万级实验记录的存储和查询

### 协作与共享

- 实验结果的团队可见性：所有实验记录对团队成员开放，避免信息孤岛
- 实验标注与评论：对实验结果添加文字评论，记录观察和假设
- 实验报告生成：从实验追踪系统中自动生成实验报告，包含参数表、指标对比图、结论
- 权限管理：公开实验 vs 私有实验，团队级别的读写权限控制

## 框架设计原理：以 MLflow 和 W&B 为例

### MLflow 的设计哲学

- 开源优先、自托管、协议驱动：定义标准的实验记录格式，不绑定特定存储后端
- MLflow Tracking 的核心概念：Experiment -> Run -> Parameter/Metric/Artifact 三层模型
- Run 的生命周期：Active -> Finished/Failed/Killed，状态转换和清理逻辑
- 不足之处：UI 功能相对简单，协作能力有限，时序指标查询性能不佳

### W&B 的设计哲学

- SaaS 优先、协作驱动：强调团队协作和可视化分析，提供云端托管服务
- W&B Run 的丰富语义：Config、Summary、History 三类数据的差异化处理
- 实验可视化能力：训练曲线实时更新、参数-指标散点图、帕累托前沿、实验比较面板
- 不足之处：对 SaaS 的依赖、数据隐私顾虑、大规模实验的成本

### 通用设计原则

- 最小侵入性：追踪 SDK 应尽量少地侵入用户代码，降低接入成本
- 结构化记录：参数和指标应存储为结构化数据，而非自由文本，以支持查询和对比
- 不可变性：实验记录一旦创建，不应被修改（允许追加评论，但不允许篡改原始数据）
- 可扩展性：支持自定义的参数类型、指标计算方式和产物格式

## 可视化建议

- **实验追踪系统架构图**：展示训练脚本 -> 追踪 SDK -> 存储后端（元数据库 + 时序库 + 对象存储）-> Web UI 的完整链路
- **实验对比面板**：多实验训练曲线叠加图，高亮最佳实验和帕累托前沿
- **参数-指标关系图**：平行坐标图（Parallel Coordinates），展示多维参数与指标的关系
- **实验时间线**：甘特图风格的时间线，展示实验的创建时间、运行时长和完成状态

## 代码示例建议

- **简易实验追踪系统**：实现 Experiment -> Run -> Parameter/Metric 的三层模型，支持参数记录、指标时序记录和实验对比查询（Python + SQLite，约 120 行）
- **训练回调集成**：实现一个通用的训练 Callback，在 epoch 结束时自动记录指标，训练结束时自动保存模型和元数据（Python，约 60 行）
- **实验对比与帕累托前沿**：从实验记录中检索多组实验的参数和指标，绘制平行坐标图和帕累托前沿（Python + matplotlib，约 60 行）
