# 特征存储

特征工程是机器学习中最具复用价值的环节。一个用户画像特征，可能同时被推荐系统、风控模型、运营分析三个团队使用；但现实中，每个团队往往各自计算、各存一份，结果不一致、逻辑不同步、计算资源浪费。特征存储（Feature Store）要解决的就是特征计算和特征消费之间的协作问题：让特征只计算一次，处处可用、版本一致、语义统一。

## 特征存储的核心问题

### 特征重复计算与不一致

- 不同团队对同一特征独立计算，逻辑可能存在细微差异（如时间窗口对齐方式、空值处理策略），导致同一特征在不同模型中含义不同
- 特征逻辑的散落：特征计算代码分散在各项目仓库中，没有统一的定义和注册机制
- 计算资源浪费：同一特征被多份 Pipeline 重复计算，尤其在大规模特征（如用户行为序列特征）上浪费严重
- 特征发现困难：新项目不知道已有哪些可用特征，只能从零开始构建

### 离线特征与在线特征的割裂

- 离线特征（Offline Features）用于模型训练，从数据仓库批量计算，延迟可以接受分钟到小时级
- 在线特征（Online Features）用于模型推理，需要毫秒级响应，通常从低延迟 KV 存储中读取
- 同一特征的离线版本和在线版本可能逻辑不同：离线用 30 天窗口计算平均消费额，在线用 7 天窗口，模型训练与推理的特征分布不一致
- 训练-推理偏差（Training-Serving Skew）的直接原因：离线和在线的特征计算逻辑未对齐

### 特征存储的定位

- 特征存储不是数据仓库的替代品，而是介于数据仓库（存储原始数据）和模型（消费特征）之间的中间层
- 三个核心职责：特征定义与注册、特征计算与调度、特征服务与交付
- 特征存储的边界：不负责原始数据的采集和存储，不负责模型的训练和推理，只负责特征的标准化管理

## 特征定义与注册

### 特征实体与特征组

- 特征实体（Entity）是特征的关联键，如 user_id、item_id、transaction_id，定义了特征的粒度
- 特征组（Feature Group）是逻辑相关的特征集合，如"用户画像特征组"包含年龄、性别、注册天数等
- 特征组的定义包含：实体键、特征列表、数据类型、计算逻辑、更新频率、存储策略
- 特征组是特征版本管理和权限控制的基本单元

### 特征元数据模型

- 特征的完整元数据：名称、描述、数据类型、所属特征组、实体键、计算逻辑描述、创建时间、最后更新时间、数据质量指标
- 特征的血缘关系：记录特征的上游数据源和下游消费者，支持影响分析
- 特征的语义标注：标注特征的业务含义、取值范围、异常值定义，降低特征使用者的理解成本
- 元数据驱动的特征发现：通过实体键和标签搜索可用特征，避免重复建设

### 特征版本与演化

- 特征逻辑变更时需要版本化：v1 的"近 7 天消费额"和 v2 的"近 7 天消费额（剔除退款）"是不同版本
- 版本兼容性：新版本是否与旧版本的输出格式兼容，下游模型是否需要重新训练
- 特征废弃流程：标记 deprecated -> 通知下游消费者 -> 设置下线时间 -> 删除
- 向后兼容的变更策略：新增特征字段不破坏旧版本，修改计算逻辑必须创建新版本

## 特征计算与调度

### 离线特征计算

- 批处理模式：定时从数据仓库读取原始数据，执行特征变换逻辑，写入特征存储
- 计算框架选择：SQL（简单聚合）、Spark（大规模分布式计算）、Python UDF（复杂逻辑）
- 增量计算 vs 全量计算：增量计算只处理新增数据，效率高但逻辑复杂；全量计算简单但代价大
- 回填（Backfill）场景：新增特征时需要计算历史数据，为已有模型提供训练样本

### 在线特征计算

- 流处理模式：实时消费消息队列中的事件，增量更新特征值，写入在线存储
- 窗口计算：滑动窗口聚合（如最近 1 小时的点击次数），需要在流处理框架中维护窗口状态
- 延迟与一致性：在线特征的计算延迟直接影响推理服务的响应时间，需要控制端到端延迟
- 降级策略：在线特征不可用时回退到最近一次的离线值，保证服务可用性

### 离线-在线一致性保证

- 一致性的核心要求：同一特征在训练时（离线）和推理时（在线）的值必须相同
- 实现路径一：共享计算逻辑，离线和在线调用同一份特征定义代码，由特征存储框架负责适配不同执行引擎
- 实现路径二：离线计算结果写入在线存储，推理时直接读取，避免在线重复计算
- 一致性校验：定期对比离线和在线特征的统计分布，检测偏差
- 框架层面的解决方案： Feast 的 Feature Server 架构、Tectonic 的统一特征计算引擎

## 特征服务

### 离线特征服务

- 训练数据生成：根据实体键列表和时间戳，从特征存储中检索对应的特征值，拼接为训练样本
- 时间旅行查询：支持查询历史时刻的特征值，保证训练数据中不会出现数据泄露（使用未来特征）
- Point-in-Time Correctness：训练样本的特征值必须是该样本对应时间点的快照，而非查询时的最新值
- 大规模检索的性能优化：列式存储 + 聚簇索引 + 分区裁剪

### 在线特征服务

- 低延迟读取：推理请求到达时，从在线存储中批量检索所需特征，延迟要求 <10ms
- 在线存储的选择：Redis（内存级延迟，成本高）、DynamoDB（SSD 延迟，成本适中）、自研 KV 存储
- 特征预热：模型上线前将所需特征批量加载到在线存储，避免冷启动时的缓存未命中
- 特征向量拼接：一次推理请求可能需要多个特征组的特征，需要跨特征组的高效拼接

### 特征服务架构

- 特征服务层作为模型推理的前置环节，屏蔽底层存储差异，对外提供统一的 gRPC/REST 接口
- 批量请求优化：将多个推理请求的特征检索合并为一次批量读取，减少存储访问次数
- 缓存策略：热点特征缓存到推理服务本地，减少对特征存储的网络访问
- 容错与降级：特征存储不可用时返回默认值或最近缓存值，不阻塞推理服务
