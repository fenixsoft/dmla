# 数据质量监控

数据是机器学习系统的燃料，燃料出了问题，再精密的引擎也无法正常运转。然而数据质量问题的隐蔽性远超想象：一个悄然发生的 Schema 变更可能让特征工程静默出错，一个传感器故障可能让 5% 的训练样本标签翻转，一个上游系统升级可能让时间戳格式从毫秒变成秒。这些问题不会抛出异常，只会让模型性能在不知不觉中衰退。数据质量监控要做的，就是在数据进入 ML Pipeline 之前，系统性地发现和拦截这些问题。

## 数据质量的核心问题

### 数据质量问题分类

- 完整性问题：缺失值、缺失行、缺失列，数据采集或传输过程中的丢失
- 一致性问题：同一实体在不同数据源中的值矛盾，跨表外键不匹配
- 准确性问题：值超出合理范围、标签错误、单位不统一
- 时效性问题：数据延迟到达、时间戳错乱、更新频率不符合预期
- 唯一性问题：重复记录导致统计偏差、模型训练样本重复

### 数据质量问题的隐蔽性

- ML 系统不会因为数据质量差而直接报错，模型照常训练，只是效果变差
- 性能下降的原因排查链条长：模型问题？特征问题？数据问题？需逐层排查
- 部分数据质量问题只在特定条件下暴露：某个特征在训练集中分布正常，在生产数据中却出现大量缺失
- 数据质量问题的累积效应：多个小问题叠加，导致模型性能骤降，但单个问题的影响难以量化

### 主动监控 vs 被动发现

- 被动发现：用户投诉、模型性能监控报警后，追溯发现是数据质量问题
- 主动监控：在数据流入 Pipeline 时自动检测异常，阻止问题数据进入训练和推理
- 主动监控的成本：需要定义质量规则、维护基线、处理误报
- 投资回报分析：数据质量问题越早发现，修复成本越低；延迟发现的成本呈指数级增长

## 数据质量规则体系

### Schema 校验

- Schema 定义数据的结构约束：列名、数据类型、nullable、取值范围、正则模式
- Schema 演化与兼容性：上游系统增加新列、修改数据类型、重命名字段时，Schema 校验应检测变更并评估影响
- 严格模式 vs 宽松模式：严格模式拒绝任何 Schema 不匹配的数据，宽松模式记录告警但允许通过
- Schema 校验的实现：JSON Schema（半结构化数据）、Protobuf（结构化数据）、自定义校验器（表格数据）

### 统计规则

- 数值型特征的统计规则：均值/中位数/标准差的合理区间、分位数约束、零值/负值比例
- 分类型特征的统计规则：取值集合约束、各类别比例约束、新增类别检测
- 时间序列的统计规则：趋势约束（不应突然增长/下降 50%）、周期性约束（周末/工作日模式）、平稳性检测
- 统计规则的设定方法：基于历史数据计算基线，设定容忍区间（如均值正负 3 倍标准差）

### 业务规则

- 跨字段一致性规则：年龄 >= 18 且职业为学生是合理的，年龄 < 10 且职业为工程师是不合理的
- 跨表关联规则：订单表中的 user_id 必须在用户表中存在，外键完整性约束
- 业务逻辑约束：消费金额 >= 0、交易时间 <= 当前时间、出生日期 <= 注册日期
- 标签质量规则：多标注者的一致性（Cohen's Kappa）、标注分布的合理性

### 规则的优先级与组合

- 规则按严重程度分级：阻断（Block，阻止数据流入 Pipeline）、告警（Warn，记录但允许通过）、信息（Info，仅记录）
- 规则组合策略：全部通过 / 任一阻断 / 加权评分 / 规则链（前序规则失败则跳过后续）
- 规则的维护成本：规则数量随数据规模增长，需要定期审查规则的适用性和误报率
- 规则引擎的设计：声明式规则定义（YAML/JSON）优于硬编码，支持热更新

## 数据质量监控架构

### 批量校验 vs 流式校验

- 批量校验：数据写入后定时扫描，适合离线训练数据的周期性检查
- 流式校验：数据流入时实时检查，适合在线推理数据的即时拦截
- 批量校验的延迟问题：从数据写入到发现问题可能有数小时延迟，问题数据已进入下游
- 流式校验的性能约束：校验逻辑需要在毫秒级完成，不能成为数据管道的瓶颈

### 基线构建与漂移检测

- 基线（Baseline）是数据质量的参考标准，通常从历史正常数据中统计得到
- 基线的维度：Schema 基线（结构）、统计基线（分布）、业务基线（规则）
- 基线的更新策略：固定基线（以某个时间点为基准）、滚动基线（最近 N 天的统计量）、自适应基线（根据趋势调整）
- 基线漂移：数据本身在正常演化（如用户年龄分布随产品定位调整），基线需要同步演化，否则产生误报

### 质量监控仪表盘

- 数据质量评分：综合各维度规则通过率，给出 0-100 的质量评分，便于跨数据集对比
- 趋势图：展示各统计指标随时间的变化趋势，辅助发现渐进式退化
- 异常事件列表：按时间倒序展示触发的规则告警，支持按严重程度过滤
- 根因分析辅助：异常事件关联上游数据源变更、Pipeline 代码变更等上下文信息

## 数据质量问题的修复

### 自动修复策略

- 缺失值填充：数值型用均值/中位数，分类型用众数/新增"未知"类别，时间序列用前向填充
- 异常值截断：超出合理范围的值截断到边界值，避免极端值影响模型训练
- 去重策略：基于主键去重，保留最新记录或最完整记录
- 自动修复的风险：修复可能掩盖更深层的数据问题，修复后的数据分布可能与原始分布不同

### 人工审核流程

- 自动修复无法处理的场景：标签错误需要人工判断、Schema 变更需要业务确认、业务规则冲突需要决策
- 审核流程设计：问题分类 -> 分配给对应负责人 -> 审核处理 -> 验证修复效果
- 审核的效率优化：相似问题批量处理、提供修复建议、自动化回归测试
- 审核记录与知识积累：每次修复的决策和原因应被记录，形成数据质量知识库

### 数据质量闭环

- 监控发现问题 -> 根因分析 -> 修复 -> 验证 -> 更新规则，形成闭环
- 闭环的关键指标：问题发现时间（MTTD）、修复时间（MTTR）、复发率
- 预防性措施：根据历史问题模式，提前在新数据源接入时设置对应规则
- 数据质量文化：质量问题不仅是工程问题，也需要组织层面的重视和流程保障

## 可视化建议

- **数据质量监控架构图**：展示数据流入 -> 规则校验 -> 结果分类（通过/告警/阻断）-> 修复/审核的完整流程
- **质量评分趋势图**：时间序列图，展示数据质量评分随时间的变化，标注异常事件和修复操作
- **统计漂移检测图**：对比当前批次与基线的特征分布叠加图（直方图或 KDE），高亮漂移区域
- **规则优先级矩阵**：二维矩阵，横轴为影响范围（单列/单表/跨表），纵轴为严重程度，展示规则的分布

## 代码示例建议

- **声明式数据质量规则引擎**：基于 YAML 定义质量规则，自动对 DataFrame 执行校验，输出通过/告警/阻断结果（Python + pandas，约 100 行）
- **统计基线构建与漂移检测**：从历史数据计算统计基线，对新批次数据进行 PSI（Population Stability Index）检测，输出漂移报告（Python + numpy，约 70 行）
- **Schema 演化检测工具**：对比两个数据集的 Schema 差异，识别新增/删除/修改的列，评估兼容性影响（Python，约 50 行）
