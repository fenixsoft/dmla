# 编排与容错

多智能体系统不是多个 Agent 的简单堆叠，而是一个需要精心编排的分布式系统。编排（Orchestration）负责协调多个 Agent 的执行流程，容错（Fault Tolerance）负责在异常发生时保障系统的可用性和正确性。这篇文章回答的核心问题是：如何编排多个 Agent 的执行流程，并在各种故障场景下保障系统的可靠性？

## 编排的基础

### 编排器的职责

- 编排器（Orchestrator）是多智能体系统的"指挥中心"，负责任务分解、Agent 分配、流程控制、结果整合
- 编排器与 Agent 的关系：编排器是管理者，Agent 是执行者。编排器不执行具体任务，但决定谁执行什么、何时执行、如何组合结果
- 编排器的核心能力：全局规划（理解整体任务和依赖关系）、动态调度（根据执行情况调整分配）、异常处理（检测和恢复故障）
- 编排器的实现形式：独立的编排服务、嵌入在协调者 Agent 中的编排逻辑、声明式的工作流定义

### 静态编排与动态编排

- 静态编排（Static Orchestration）在任务开始前确定完整的执行流程，运行时严格按流程执行
- 静态编排的表示：工作流定义（DAG 或状态机），每个节点是一个 Agent 任务，边是依赖关系或转移条件
- 静态编排的优势：流程可预测、可审计、可复现；劣势：缺乏灵活性，无法应对意外情况
- 动态编排（Dynamic Orchestration）在运行时根据执行结果和环境变化决定下一步，流程不预先确定
- 动态编排的实现：编排器作为 Agent，根据当前状态推理下一步应该做什么、分配给哪个 Agent
- 动态编排的优势：灵活适应变化；劣势：不可预测、难以审计、行为可能不一致
- 混合编排：框架定义流程骨架（哪些阶段必须经过），细节由编排器动态填充（每个阶段内如何执行）

### 工作流定义

- 工作流（Workflow）是对执行流程的形式化描述，是静态编排的核心
- 工作流的表示方式：DAG（有向无环图，表示任务依赖和执行顺序）、状态机（有限状态自动机，表示状态转移和条件）、Petri 网（并发和同步的精确建模）
- 工作流的关键元素：任务节点（Agent 执行的工作）、控制节点（条件判断、循环、并行分支）、数据边（任务之间的数据流）、事件触发（外部事件驱动的流程启动）
- 工作流的可视化：流程图是最直观的工作流表示，便于人类理解和审查

## 编排模式

### 管道编排

- 管道编排（Pipeline Orchestration）将任务组织为线性流水线，每个 Agent 处理一个阶段，输出传递给下一个 Agent
- 管道编排的典型场景：数据处理流水线（采集 → 清洗 → 分析 → 可视化）、内容生产流水线（调研 → 撰写 → 审校 → 发布）
- 管道编排的优势：流程清晰、每个 Agent 职责单一、容易调试
- 管道编排的局限：吞吐量受最慢阶段限制（木桶效应）、一个阶段失败整个流水线停滞、不支持条件分支

### 扇出-扇入编排

- 扇出-扇入（Fan-Out/Fan-In）编排将一个任务分解为多个并行子任务，分配给多个 Agent 同时执行，然后汇总结果
- 扇出阶段：将任务分解并分发，每个 Agent 独立执行自己的子任务
- 扇入阶段：等待所有 Agent 完成，汇总结果，可能需要冲突解决或结果融合
- 扇出-扇入的典型场景：多方案评估（多个 Agent 各自提出方案，汇总后选择最优）、多文件处理（每个 Agent 处理一个文件，最后合并）
- 扇入的挑战：部分 Agent 可能失败或超时，需要决定是等待还是用部分结果继续

### 条件路由编排

- 条件路由（Conditional Routing）根据中间结果选择不同的执行路径，实现动态分支
- 条件路由的表示：在工作流 DAG 中加入条件判断节点，根据判断结果选择后续路径
- 条件路由的典型场景：代码审查（测试通过则自动合并，测试失败则通知开发者修复）、客户服务（简单问题自动回答，复杂问题转人工）
- 条件路由的挑战：条件判断本身可能出错（误判导致走错路径）、路径数量增多时工作流复杂度急剧上升

### 递归编排

- 递归编排（Recursive Orchestration）允许编排器将子任务再次分解和编排，形成嵌套的编排层次
- 递归编排的典型场景：大型项目分解为模块，每个模块再分解为子任务，每个子任务可能还需要进一步分解
- 递归的终止条件：任务粒度足够小，可以由单个 Agent 直接执行；或达到最大递归深度
- 递归编排的挑战：深度递归可能导致巨大的编排开销、调试困难（需要追踪多层嵌套的执行过程）

## 容错机制

### 故障类型

- Agent 故障：Agent 崩溃、超时、返回错误结果。原因可能是 LLM API 限流、上下文溢出、幻觉导致无效操作
- 工具故障：工具调用失败、返回异常结果、响应超时。原因可能是外部服务不可用、参数错误、权限不足
- 通信故障：消息丢失、延迟、乱序。原因可能是网络问题、消息队列溢出、序列化错误
- 编排故障：编排器决策错误、工作流定义有缺陷、状态管理不一致
- 外部环境故障：数据源变化、API 接口变更、资源耗尽

### 超时与重试

- 超时（Timeout）是容错的第一道防线：为每个操作设置合理的超时时间，超时后视为失败
- 超时时间的设置：太短导致正常操作被误判为失败，太长导致故障影响扩大。应根据操作类型设置不同超时（工具调用 30 秒、Agent 推理 5 分钟、整体任务 1 小时）
- 重试（Retry）是最基本的恢复策略：操作失败后重新执行，假设失败是暂时性的
- 重试策略：固定间隔重试、指数退避重传（Exponential Backoff，每次重试间隔加倍）、抖动退避（在退避间隔上加入随机抖动，避免多个重试同时发生）
- 重试的局限：不是所有失败都适合重试。确定性错误（如参数格式错误）重试不会成功，副作用操作（如发送邮件）重试可能导致重复

### 降级与兜底

- 降级（Degradation）是在主要方案失败时切换到备选方案，以降低质量换取可用性
- 降级策略的层次：工具降级（主工具不可用时切换到备选工具）、Agent 降级（专业 Agent 不可用时切换到通用 Agent）、任务降级（完整任务无法完成时完成部分任务）
- 兜底（Fallback）是最后的保障：当所有自动恢复策略都失败时，将任务转交给人类处理
- 降级与兜底的设计原则：每个关键操作都应有至少一个备选方案，备选方案的质量可以低于主方案，但不能完全缺失

### 检查点与恢复

- 检查点（Checkpoint）是保存系统在某个时刻的完整状态，以便在故障后从检查点恢复执行
- 检查点的内容：已完成的任务及其结果、当前正在执行的任务及其进度、工作流的状态（哪些节点已完成、哪些正在执行）
- 检查点的粒度：任务级检查点（每个任务完成后保存）、阶段级检查点（每个阶段完成后保存）、全局检查点（整个工作流的完整快照）
- 恢复策略：从最近的检查点重新开始、跳过已完成的任务从失败点继续、完全重新执行
- 检查点的代价：保存状态需要存储空间和 I/O 开销，粒度越细开销越大

### 断路器模式

- 断路器（Circuit Breaker）模式在检测到某个组件持续失败时，暂时停止向其发送请求，避免故障扩散
- 断路器的三个状态：关闭（正常工作，请求正常通过）、打开（检测到持续失败，请求被直接拒绝）、半开（试探性恢复，允许少量请求通过以检测是否恢复）
- 断路器在 Agent 系统中的应用：当某个工具持续返回错误时，断路器打开，编排器自动切换到备选工具或降级方案
- 断路器的参数：失败阈值（连续失败多少次后打开断路器）、恢复超时（打开状态持续多久后进入半开状态）、半开试探量（半开状态允许通过的请求数量）

## 一致性与幂等性

### 操作的幂等性

- 幂等性（Idempotency）是指同一操作执行一次和执行多次的效果相同。在可能重试的系统中，幂等性是正确性的保障
- 幂等操作的例子：读取数据（读多次结果相同）、设置值（设为 5 多次结果仍是 5）、删除已删除的记录
- 非幂等操作的例子：追加数据（追加多次会产生重复）、发送邮件（发送多次会收到多封）、递增计数器
- 将非幂等操作转化为幂等操作的方法：使用唯一请求标识（去重）、条件更新（只在满足条件时执行）、记录已执行的操作（避免重复执行）

### 结果一致性

- 多个 Agent 并行执行时，可能产生不一致的结果。一致性（Consistency）是确保系统状态始终满足特定约束的性质
- 一致性级别：强一致性（任何时刻所有 Agent 看到的状态一致）、最终一致性（允许暂时不一致，但最终会收敛）、因果一致性（有因果关系的操作顺序一致）
- Agent 系统中通常选择最终一致性：强一致性的协调开销太大，而 Agent 任务通常可以容忍暂时的不一致
- 一致性冲突的解决：时间戳优先（后写入覆盖先写入）、来源优先（高优先级 Agent 的结果覆盖低优先级）、合并（将冲突的结果合并为兼容的版本）

### 补偿事务

- 补偿事务（Compensating Transaction）是在无法原子回滚时，通过执行反向操作来撤销已完成的步骤
- 补偿事务与数据库回滚的区别：数据库回滚是原子的、自动的，补偿事务是语义的、需要手动设计
- 补偿事务的例子：已发送的邮件无法撤回，但可以发送一封"请忽略上一封邮件"的补偿邮件
- 补偿事务的设计原则：每个有副作用的操作都应设计对应的补偿操作，补偿操作本身也应是幂等的

## 可观测性

### 日志与追踪

- 日志（Logging）是记录系统行为的基本手段，每个 Agent 的决策、工具调用、状态变化都应有日志记录
- 分布式追踪（Distributed Tracing）在多 Agent 系统中尤为重要：一个请求可能经过多个 Agent，需要端到端的追踪来理解完整路径
- 追踪的核心概念：Trace（一次完整请求的追踪）、Span（一个 Agent 或操作的追踪片段）、Context（跨 Agent 传递的追踪上下文）
- 日志与追踪的平衡：过于详细的日志产生大量存储和 I/O 开销，过于简略的日志无法支持有效调试

### 指标与告警

- 关键指标（Metrics）：任务成功率、平均执行时间、Agent 利用率、工具调用失败率、消息延迟
- 告警（Alerting）：当指标超过阈值时触发告警，如任务失败率超过 10%、平均执行时间超过预期的 2 倍
- 指标的聚合粒度：实时（秒级，用于即时告警）、短期（分钟级，用于趋势分析）、长期（小时/天级，用于容量规划）
- 指标驱动的自适应：根据指标动态调整系统参数，如根据 Agent 利用率调整并行度、根据失败率调整重试策略
