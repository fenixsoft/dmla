# 从 LLM 到 Agent

语言模型是一个被动的文本生成器——给它一段输入，它输出一段文本，任务就结束了。而智能体（Agent）是一个自主的决策系统——它接收目标，自主规划步骤、调用工具、观察结果、修正行动，直到目标达成。从 LLM 到 Agent 的跨越，本质上是把"单轮问答"升级为"多轮循环决策"。这篇文章回答的核心问题是：一个只会生成文本的模型，如何变成一个能自主完成复杂任务的系统？

## LLM 的能力边界

### 生成式模型的被动性

- LLM 的基本运作模式是"输入文本 → 输出文本"，模型本身没有行动能力，无法影响外部世界
- 模型输出的是 token 序列，而非操作指令。说"帮我发一封邮件"和真正发出一封邮件之间，存在行动鸿沟
- 模型的知识是静态的，截止于训练数据的末尾。对于训练后发生的事件、私有数据库中的信息、实时变化的数据，模型无从得知
- 被动性并非缺陷，而是设计选择。模型专注于语言理解与生成，行动能力通过外部机制补全

### 上下文窗口的局限

- 即使是百万 token 的上下文窗口，也无法容纳长时任务中的全部信息。一个持续数小时的编码任务，中间产生的文件内容、终端输出、调试日志会迅速消耗上下文空间
- 上下文窗口的"遗忘"是硬性的——超出窗口的信息对模型而言完全不存在，不同于人类的模糊记忆
- 上下文窗口的利用效率随长度增加而下降，长上下文中的信息检索准确率会降低（"迷失在中间"现象）
- 这一局限直接催生了对外部记忆系统的需求，这是 Agent 架构的必要组件之一

### 幻觉与不可靠性

- 幻觉（Hallucination）是 LLM 生成看似合理但不符合事实的内容的倾向，其根源在于模型优化的是文本流畅性而非事实准确性
- 在 Agent 场景中，幻觉的后果远比聊天场景严重：聊天时编造一个不存在的餐馆只是尴尬，Agent 编造一条不存在的 API 调用则会导致任务失败
- 幻觉无法被彻底消除，但可以通过工具验证、自我反思、外部校验等 Agent 机制来检测和修正
- 理解幻觉的存在，是设计 Agent 容错机制的前提

## Agent 的核心架构

### 感知-决策-行动循环

- Agent 的基本运行模式是 OODA 循环（Observe-Orient-Decide-Act）：观察环境状态、理解当前处境、决定下一步行动、执行行动并观察结果
- LLM 在循环中扮演"决策"角色：接收观察信息，生成下一步行动的描述，由执行引擎将描述转化为实际操作
- 循环的核心挑战是"停止条件"：Agent 需要知道何时任务已完成，否则会陷入无限循环或过早终止
- OODA 循环与强化学习中的 Agent-Environment 交互模式本质相同，区别在于 LLM Agent 使用语言而非数值向量作为状态和动作的表示

### ReAct 模式

- 2022 年，姚顺雨（Shunyu Yao）在论文《ReAct: Synergizing Reasoning and Acting in Language Models》中提出 ReAct 模式，将推理（Reasoning）与行动（Acting）交织进行
- ReAct 的工作流程：思考（Thought）→ 行动（Action）→ 观察（Observation）→ 思考 → 行动 → 观察 → ...
- 思考步骤让模型显式推理当前状态和下一步策略，行动步骤让模型调用工具与外部世界交互，观察步骤将工具返回的结果纳入推理上下文
- ReAct 相比纯推理（Chain-of-Thought）或纯行动（Act-only）的优势：推理帮助模型制定更合理的行动计划，行动帮助模型获取推理所需的实时信息
- ReAct 的局限：每次循环都消耗上下文空间，长任务中早期信息可能被挤出窗口；思考步骤的 token 开销增加了推理成本

### 规划器-执行器分离

- ReAct 模式是"边想边做"，而规划器-执行器（Planner-Executor）模式是"先想后做"：先由规划器生成完整的行动方案，再由执行器逐步执行
- 规划器的职责是生成任务分解和执行顺序，执行器的职责是根据规划逐步执行并在异常时上报
- 两种模式不是互斥的，实践中常采用混合策略：先规划整体方案，执行中遇到意外时重新规划
- 分离的好处是规划与执行可以独立优化，规划的粒度可以调整，执行过程可以被监控和中断
- 分离的代价是增加了系统复杂度，规划器与执行器之间需要通信协议，且规划可能因信息不完整而失效

## 从对话到自主行动

### 目标理解与任务分解

- Agent 接收的通常是模糊的高层目标（如"帮我部署一个网站"），需要将其分解为可执行的子任务序列
- 任务分解的粒度是关键权衡：粒度太粗，子任务仍需复杂推理；粒度太细，规划开销过大且丧失灵活性
- 分解策略的层次：顶层分解将大目标拆为子目标，子目标再分解为具体步骤，形成任务树
- 常见的分解启发式：按时间顺序（先做什么后做什么）、按依赖关系（哪些步骤可以并行）、按资源约束（哪些步骤需要特定工具）

### 执行监控与自我修正

- Agent 执行过程中可能遇到各种意外：工具调用失败、返回结果与预期不符、外部环境发生变化
- 自我修正（Self-Correction）能力是 Agent 区别于简单脚本的核心特征：Agent 能检测到执行偏差，分析原因，并调整后续行动
- 修正策略的层次：重试（同一操作再试一次）、换路（选择替代方案）、回溯（回到之前的决策点重新规划）、升级（请求人类协助）
- 修正的挑战在于判断何时该修正、何时该坚持。过度修正会导致任务永远无法完成，修正不足则会在错误路径上越走越远

### 自主性的等级

- 自主性（Autonomy）不是非此即彼，而是一个从完全手动到完全自主的光谱
- L0-无自主性：人类给出每一步指令，LLM 只是执行工具调用
- L1-建议自主性：LLM 提出行动建议，人类确认后执行
- L2-监督自主性：LLM 自主执行常规操作，遇到不确定的情况时请求人类确认
- L3-条件自主性：LLM 在预设边界内完全自主执行，仅在超出边界时暂停
- L4-完全自主性：LLM 自主完成所有操作，人类仅设定目标
- 自主性等级的选择取决于任务的容错率和错误后果的严重性。写代码可以高自主性，删除数据库则需要低自主性

## Agent 的设计原则

### 单一职责与组合

- 一个 Agent 应该有明确的职责边界，而不是试图完成所有任务
- 单一职责的 Agent 更容易测试、调试和复用，组合多个专精 Agent 比构建一个万能 Agent 更可行
- 组合的方式：串联（Agent A 的输出是 Agent B 的输入）、并行（多个 Agent 同时处理不同子任务）、层级（管理者 Agent 分配任务给工人 Agent）

### 最小权限原则

- Agent 应只获得完成任务所需的最小权限，而非系统全权
- 工具调用的权限控制：只暴露必要的工具，敏感操作（如文件删除、网络请求）需要额外授权
- 权限与自主性的关系：权限越高，自主性越需要受限；自主性越高，权限越需要约束

### 可观测性与可中断性

- Agent 的每一步行动都应有日志记录，包括决策理由、工具调用参数和返回结果
- 人类应能在任意时刻中断 Agent 的执行，这是安全底线
- 可观测性不仅是安全需求，也是调试需求。Agent 行为异常时，需要追溯是哪一步决策出了问题

### 优雅降级

- Agent 在工具不可用、上下文溢出、API 限流等异常情况下，应能优雅降级而非崩溃
- 降级策略：使用备选工具、缩小任务范围、请求人类介入
- 降级设计要求在正常流程之外，为每个可能的失败点准备备选方案
