# 规划与推理

Agent 面对的现实任务往往不是一步能完成的。部署一个应用需要规划步骤，调试一段代码需要分析因果，撰写一份报告需要组织素材。这些任务的共同特征是需要多步推理和前瞻性规划。这篇文章回答的核心问题是：Agent 如何将复杂目标分解为可执行的步骤序列，并在执行过程中动态调整？

## 规划的基础

### 任务分解

- 任务分解是将高层目标拆解为可执行的子任务序列的过程，是 Agent 规划的起点
- 分解的粒度权衡：粗粒度分解减少规划步数但增加单步复杂度，细粒度分解降低单步难度但增加协调开销
- 分解策略的类型：按时间顺序分解（线性流程）、按依赖关系分解（DAG 结构）、按功能模块分解（层级结构）
- 好的分解满足 MECE 原则（Mutually Exclusive, Collectively Exhaustive）：子任务之间不重叠，合起来覆盖全部目标
- 分解的常见失败模式：遗漏关键步骤、子任务之间隐含依赖未被识别、粒度不均匀导致某些子任务仍然过于复杂

### 计划表示

- 计划（Plan）需要一种机器可读的表示形式，以便 Agent 执行和监控系统解析
- 线性计划：简单的步骤列表，适合顺序执行的任务。表示为 `[step1, step2, ..., stepN]`
- 条件计划：包含分支判断的计划，根据中间结果选择不同路径。表示为带条件节点的有向图
- 层级计划：树状结构，每个节点可以展开为子计划。顶层是粗粒度目标，逐层展开为细粒度步骤
- 计划表示的选择取决于任务的复杂度和不确定性：确定性任务用线性计划，不确定性高的任务用条件计划

### 依赖分析与调度

- 子任务之间的依赖关系决定了执行顺序：如果 B 依赖 A 的输出，则 A 必须在 B 之前完成
- 依赖图（Dependency Graph）是有向无环图（DAG），节点是子任务，边是依赖关系
- 基于依赖图的调度：拓扑排序确定执行顺序，无依赖的节点可以并行执行
- 隐式依赖的识别是难点：两个任务看似独立，但可能通过共享资源产生隐性冲突（如都写同一文件）

## 推理策略

### 前向推理

- 前向推理（Forward Reasoning）是从已知条件出发，逐步推导到目标。思路是"我知道什么，能推出什么"
- 在 Agent 场景中的表现：从当前状态出发，选择最有可能接近目标的下一步行动
- 前向推理的优势：每一步都有充分依据，推理过程可靠
- 前向推理的局限：缺乏目标导向，可能走入与目标无关的分支；在状态空间大时，搜索效率低

### 目标回推

- 目标回推（Backward Reasoning）是从目标出发，逆向推导需要满足的前置条件。思路是"要达到目标，需要什么"
- 在 Agent 场景中的表现：先明确最终目标，再反推需要完成的子目标，直到所有子目标都有已知的方法实现
- 目标回推的优势：目标导向，每一步推理都服务于最终目标，不会浪费计算在无关路径上
- 目标回推的局限：需要明确的目标定义，对模糊目标效果差；当前置条件不唯一时，搜索空间可能爆炸

### 混合推理

- 实际 Agent 任务通常需要前向推理与目标回推的结合：用目标回推确定大方向和关键里程碑，用前向推理填充具体步骤
- 混合推理的典型模式：先回推确定子目标树，再前向执行每个子目标的具体步骤
- 何时回推、何时前推的判断本身是一个元推理问题，模型需要根据任务特征自主选择推理方向

## 规划算法

### 树搜索

- 将规划问题建模为搜索问题：每个节点是一个状态，每条边是一个行动，目标是找到从初始状态到目标状态的路径
- 广度优先搜索（BFS）保证找到最短路径但空间消耗大，深度优先搜索（DFS）空间效率高但可能陷入无限分支
- 启发式搜索（如 A*）使用启发函数评估节点的"前景"，优先搜索更有希望的路径，在状态空间大时效率显著优于盲目搜索
- LLM Agent 中的树搜索：模型充当启发函数，评估每个候选行动的可行性，剪枝低概率分支

### 蒙特卡洛树搜索

- 蒙特卡洛树搜索（MCTS）在搜索空间巨大且评估函数不精确时表现优异，已被应用于 LLM 推理规划
- MCTS 的四个步骤：选择（Select，沿树选择最有前景的节点）、扩展（Expand，扩展一个新子节点）、模拟（Rollout，从新节点快速模拟到终局）、回传（Backpropagate，将模拟结果反向传播更新节点价值）
- 2024 年，研究者将 MCTS 与 LLM 结合：LLM 负责生成候选行动和评估状态价值，MCTS 负责搜索策略，两者协同提升规划质量
- MCTS 的代价是计算开销：每次规划都需要大量模拟，适合高价值、低频次的决策场景

### 自适应规划

- 自适应规划（Adaptive Planning）是指 Agent 在执行过程中根据观察结果动态调整计划，而非严格遵循预设方案
- 触发重新规划的条件：工具调用失败、观察到与预期不符的结果、发现新的信息改变了问题理解
- 重新规划的策略：局部调整（修改当前步骤的参数或替换工具）、全局重规划（放弃当前计划，从当前状态重新规划）、回溯重规划（回到之前的某个决策点，从那里重新规划）
- 自适应规划的挑战是平衡坚持与灵活：过于频繁地重新规划会导致任务永远无法完成（规划振荡），过于坚持则可能在错误路径上浪费资源

## 规划的可靠性

### 计划验证

- 生成的计划在执行前应经过验证，检查其完整性和正确性，避免在执行阶段才发现计划有误
- 验证维度：完整性（是否遗漏关键步骤）、可行性（每个步骤是否有对应的工具和能力实现）、一致性（步骤之间是否存在矛盾）
- 验证方式：模型自验证（让 LLM 审查自己生成的计划）、规则检查（用预定义规则检查常见错误）、模拟执行（在沙箱中预演计划）

### 幻觉规划检测

- LLM 生成的计划可能包含"幻觉步骤"：引用不存在的工具、假设不可能的前提、规划逻辑上不可达的目标
- 幻觉规划的检测方法：工具存在性检查（计划中引用的工具是否真实可用）、依赖可达性检查（步骤的输入是否有来源）、常识性校验（步骤是否违反基本逻辑）
- 检测本身也可能出错（假阳性：正确的计划被误判为幻觉），因此检测结果应作为预警而非阻断

### 计划的鲁棒性

- 鲁棒性（Robustness）是指计划在遇到意外情况时仍能达成目标的能力
- 提升鲁棒性的策略：为关键步骤准备备选方案、在计划中预留缓冲（如额外的验证步骤）、设置合理的超时和重试机制
- 鲁棒性与效率的权衡：鲁棒的计划通常更长更保守，执行效率可能低于乐观的计划
