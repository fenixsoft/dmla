# 检索质量：评估与优化

1966 年，英国克兰菲尔德航空学院的图书管理员西里尔·克莱弗登（Cyril Cleverdon）做了一件当时看来颇为激进的事：他设计了一套包含查询、文档和人工相关性判断的测试集，用这套数据系统性地对比了不同索引方法（作者索引、标题索引、受控词表等）的检索效果。这个后来被称为"克兰菲尔德实验"（Cranfield Experiments）的项目奠定了信息检索评估的基本范式：用标准化的查询集、文档集和相关性判断（relevance judgments）来度量检索质量，而不是依赖用户的主观反馈。半个多世纪后的今天，从 Google 到 RAG 系统，我们对检索质量的评估依然遵循着克莱弗登当年建立的"先有标准答案，再算差距"的核心思路。

前面章节介绍了向量索引结构与相似度计算，这些是检索的"基础设施"。基础设施搭建完成后，自然会问两个问题：检索效果好不好？不好时怎么改？本文用这两个问题组织全篇：前半部分讲评估，如何用量化指标诊断检索质量；后半部分讲优化，如何通过混合检索与重排序让检索结果更准确。评估是诊断的手段，优化是治疗的方案，两者构成检索系统迭代的闭环。

正式开始前先明确一点。检索评估与模型评估（如分类准确率、交叉熵损失）最大的不同在于：检索的"正确答案"通常是模糊的。判断一篇文章是否与用户查询"相关"本身带有主观性，同一个查询在不同场景下可能需要完全不同的文档。这意味着检索评估指标的设计不仅要考虑数学上的严谨性，还要考虑标注成本和实用性。理解了这一前提，就能明白为什么检索领域会有那么多指标，每种指标都在精确性、标注成本和实际业务需求之间做了不同的取舍。

## 检索评估指标

检索评估指标大致分两个层次：集合级指标关心"该找的有没有找到以及找来的对不对"，排序级指标则进一步关心"找到的文档排在第几位"。两个层次各有应用场景，但背后的设计逻辑相通。先看集合级指标。

### 集合级指标：召回率与精确率

**召回率**（Recall）和**精确率**（Precision）是检索评估中最基础的两个指标。它们的思路与分类问题中的混淆矩阵一脉相承：把检索返回的文档看作"预测为相关"，把未返回的文档看作"预测为不相关"，检索评估本质上就是在计算查全率和查准率。

以 RAG 场景为例，假设用户查询"Transformer 架构中的位置编码有哪些类型"，知识库中有 10 篇相关文档。检索系统返回了 Top-5 结果，其中 3 篇确实相关（正余弦位置编码、可学习位置编码、相对位置编码），另外 2 篇是讲注意力机制的（不相关）。此时召回率和精确率分别为：

$$\text{Recall}@5 = \frac{|\text{相关文档} \cap \text{检索结果前5}|}{|\text{相关文档}|} = \frac{3}{10} = 0.3$$

$$\text{Precision}@5 = \frac{|\text{相关文档} \cap \text{检索结果前5}|}{5} = \frac{3}{5} = 0.6$$

召回率 30% 意味着还有 70% 的相关文档没有检索到。在 RAG 场景中，召回率低意味着关键信息被遗漏，模型生成时缺少必要依据，可能产生错误的输出。精确率 60% 意味着返回的 5 篇文档中有 2 篇不相关，这些无关文档不仅浪费了上下文窗口的 Token 配额，还可能干扰 LLM 的判断，使其偏离正确的回答方向。

召回率和精确率之间存在天然矛盾。如果把检索返回数量 $k$ 从 5 扩大到 20，召回率大概率会上升（覆盖了更多相关文档），但精确率会下降（引入了更多噪声）。它们之间不是"一个变好一个变差"的简单权衡，而是检索系统在**覆盖度**和**信号纯度**之间的根本张力。当需要一个单一数值来综合衡量时，F1 分数取两者的调和平均，调和平均的特点是偏向较小的那个值，如果召回率很高但精确率很低，F1 也会被拉低，从而避免"只优化一个指标"的投机行为：

$$F1 = \frac{2 \times \text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

### 排序级指标：MRR、MAP、NDCG

集合级指标只看"文档是否被检索到"，完全不关心排序位置。但实际上，排在前面的文档对 RAG 生成的影响远大于排在后面的。大量研究表明，LLM 对上下文开头部分更敏感，这和"Lost in the Middle"现象一致：模型倾向于关注提示词开头和结尾的信息，中间部分容易被忽略。排序级指标弥补了这一缺陷，它们不仅衡量"找到了什么"，还评估"好东西有没有排在前面"。

**MRR**（Mean Reciprocal Rank，平均倒数排名）是最简单的排序指标，它只关心第一个相关文档的位置。假设有三个查询，第一个相关文档分别排在第 1 位、第 3 位和第 5 位，那么 MRR 为：

$$\text{MRR} = \frac{1}{|Q|} \sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i} = \frac{1}{3} \left( \frac{1}{1} + \frac{1}{3} + \frac{1}{5} \right) \approx 0.51$$

这个公式看着抽象，拆开来看含义很直观：

- $\text{rank}_i$ 是第 $i$ 个查询的第一个相关文档的排名位置，数字越小（越靠前）越好
- $\frac{1}{\text{rank}_i}$ 是排名的倒数，排名第 1 得 1 分，排名第 10 只得 0.1 分，排名越靠后分数衰减越快
- $\frac{1}{|Q|}$ 将所有查询的得分取平均，得到系统的整体表现
- 整体可以理解为：系统能不能把正确答案放在用户一眼就能看到的位置

MRR 的适用场景是用户只需要一个正确答案（如问答系统中"法国的首都是什么"），第一个相关结果排在第 1 位得满分，排在第 3 位得 1/3 分。它的局限也很明显：如果查询"Transformer 的优点有哪些"有 5 篇相关文档，MRR 完全不关心第 2 到第 5 篇的位置。

**MAP**（Mean Average Precision，平均精度均值）弥补了 MRR 的这个不足，它考虑所有相关文档的位置。MAP 的计算思路是：在每个相关文档出现的位置上，计算一次此时的精确率，然后取平均。举例来说，假设某查询有 3 篇相关文档，它们在检索结果中分别排在第 1、第 4 和第 7 位。AP（Average Precision）的计算过程为：

- 第 1 位：找到第 1 篇相关文档，此时精确率 = 1/1 = 1.0
- 第 4 位：找到第 2 篇相关文档，此时精确率 = 2/4 = 0.5
- 第 7 位：找到第 3 篇相关文档，此时精确率 = 3/7 ≈ 0.43
- AP = (1.0 + 0.5 + 0.43) / 3 ≈ 0.64

用公式表达：

$$\text{AP} = \frac{1}{|\text{相关文档}|} \sum_{k=1}^{n} P(k) \times \text{rel}(k)$$

拆解来看：

- $P(k)$ 是前 $k$ 个结果的精确率，也就是截至第 $k$ 位时，找到的"相关文档数 / $k$"
- $\text{rel}(k)$ 是一个指示函数：第 $k$ 位文档是相关的则为 1，不相关则为 0，它保证了只在相关文档出现的位置上记录精确率
- 分母 $|\text{相关文档}|$ 是相关文档总数，对多次精确率求和后取平均
- 整体可以理解为：相关文档出现得越靠前，各个位置上的精确率就越高，AP 就越高

MAP 就是将多个查询的 AP 取平均：$\text{MAP} = \frac{1}{|Q|} \sum \text{AP}$。它与 MRR 的核心区别在于：MRR 只看第一个相关文档的位置，MAP 则看所有相关文档的排序位置。

**NDCG**（Normalized Discounted Cumulative Gain，归一化折损累积增益）进一步支持多级相关性标注。MRR 和 MAP 都假设相关性是二值的（相关 / 不相关），但实际上有些结果"高度相关"，有些只是"部分相关"。NDCG 允许标注人员给出 0～4 或 0～3 等级别的多级标注，并通过折损因子让排序靠后的文档权重递减。

以三个文档为例，假设它们的相关性标注分别为 3（高度相关）、2（相关）、0（不相关），排在检索结果第 1、2、3 位：

$$\text{DCG}@3 = \frac{2^{3} - 1}{\log_2(1 + 1)} + \frac{2^{2} - 1}{\log_2(2 + 1)} + \frac{2^{0} - 1}{\log_2(3 + 1)} = 7 + \frac{3}{1.58} + 0 = 8.90$$

公式本身：

$$\text{DCG}@k = \sum_{i=1}^{k} \frac{2^{\text{rel}_i} - 1}{\log_2(i + 1)}$$

拆开来看：

- $\text{rel}_i$ 是排在第 $i$ 位的文档的相关性等级，$2^{\text{rel}_i} - 1$ 将等级指数放大，等级为 3 的文档贡献 7 个单位，等级为 1 的仅贡献 1 个单位，等级差距被放大
- $\frac{1}{\log_2(i+1)}$ 是折损因子：排在第 1 位除 1，第 3 位除 2，第 100 位除 6.6，模拟了用户注意力从前往后递减的自然规律
- DCG 本身不归一化，不同查询的 DCG 无法直接比较，因此 NDCG 将 DCG 除以理想排序下的 DCG（IDCG）：

$$\text{NDCG}@k = \frac{\text{DCG}@k}{\text{IDCG}@k}$$

在刚才的例子中，理想排序是将相关文档排在前面（3、2、0），IDCG@3 = (7/1) + (3/1.58) + (0/2) = 8.90，在这个排列已经是最优的情况下 NDCG = 1.0。如果实际排序是（2、0、3），DCG = 3/1 + 0 + 7/2 = 6.5，NDCG ≈ 0.73。NDCG 越接近 1，说明排序越接近理想状态。

### 指标选择指南

面对这五种指标，应该根据具体的业务场景来选择。RAG 场景最应关注 Recall@k：遗漏关键文档意味着生成模型手中缺少关键证据，输出质量从根源上就无法保障。如果同时需要关注排序质量（比如 Top-3 文档的排序对生成质量有显著影响），NDCG 比 MAP 更灵活，因为它支持多级标注。如果只需知道"第一个正确答案排第几"，MRR 就是最直接的选择。

| 指标 | 相关性标注 | 关注点 | 典型场景 |
|-----|-----------|-------|---------|
| Recall@k | 二值 | 覆盖度 | RAG 检索 |
| Precision@k | 二值 | 准确度 | 结果筛选 |
| MRR | 二值 | 首位准确 | 问答系统 |
| MAP | 二值 | 排序质量 | 文档检索 |
| NDCG@k | 多级 | 排序质量 | 搜索引擎 |

### 评估数据集

指标计算需要"标准答案"：一组查询和对应的相关性标注（relevance judgments）。标注数据的构建本身就是一个工程挑战。查询集需要覆盖不同类别（事实型、推理型、对比型）和不同难度等级，否则评估结果可能掩盖系统在特定查询类型上的弱点。相关性标注方面，二值标注（相关 / 不相关）标注简单但损失信息，多级标注（高度相关 / 相关 / 弱相关 / 不相关）更精细但标注成本高且标注者之间的一致性不易保证。一种折中方案是使用 LLM 生成初始标注再由人工审核修正：LLM 对相关性的判断与人类标注的一致性已经相当高，可以大幅降低标注成本，而人工抽检可以确保边界案例不致偏差。

社区提供了多个标准化基准，避免了每个团队都从头标注数据的重复劳动。MS MARCO 是目前最广泛使用的段落检索基准，包含百万级真实 Bing 查询和人工标注的段落相关性。BEIR 则是跨领域零样本评估的代表，覆盖了生物医学、社交媒体、学术论文等 18 个不同领域的数据集，适合评估检索模型的泛化能力。中文场景可以关注 C-MTEB，它涵盖了检索、聚类、语义相似度等多个任务，是中文嵌入模型评估的主要参考。

## 混合检索

上一节的指标告诉我们检索质量"好不好"，但如果结论是"不够好"，下一步就是改进。提升检索质量最直接的手段之一是混合检索，把两种思路完全不同的检索方法组合起来，让它们的长处互相弥补短处。

### 单一检索的局限

现代检索系统主要使用稠密检索（dense retrieval），即用嵌入模型将查询和文档分别编码为向量，通过余弦相似度计算匹配得分。稠密检索的优势在于语义泛化：向量空间中将"电动汽车"映射到与"新能源车"邻近的位置，因此即使文档中没有出现完全相同的词，语义相似的文档也能被检索到。但这种泛化能力本身就是一把双刃剑，"RFC 7231"在向量空间中可能被编码得与"RFC 7230"非常接近，因为两个 token 周围的上下文分布几乎相同，模型很难区分一个编号的细微差异。

稀疏检索（sparse retrieval）恰好相反。BM25 这类方法建立在倒排索引之上，直接匹配词项，因此 RFC 编号、API 名称、产品型号这类精确词汇永远不会被"语义近似"所干扰。但代价是"汽车"和"轿车"对 BM25 来说是完全不同的两个词，语义等价被遗失。混合检索的核心动机就在于此：稠密检索负责"理解意思"，稀疏检索负责"匹配字面"，两者各司其职。

### 稀疏检索：BM25

BM25 的全称是"Okapi BM25"，由英国伦敦城市大学的斯蒂芬·罗伯逊（Stephen Robertson）和凯伦·斯帕克·琼斯（Karen Spärck Jones）等人在 20 世纪 90 年代提出。它的前身可以追溯到斯帕克·琼斯 1972 年提出的 IDF（逆文档频率）概念，她在信息检索领域的开创性贡献为她赢得了 2004 年 ACL 终身成就奖。BM25 是 TF-IDF 的改进版本，核心改进在于引入了文档长度归一化和词频饱和控制，让评分体系更符合实际检索中的直觉：一个词出现一次很重要，出现两次更重要，但出现一百次并不会比出现十次重要一百倍。

$$\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t, d) \cdot (k_1 + 1)}{f(t, d) + k_1 \cdot (1 - b + b \cdot |d| / \text{avgdl})}$$

这个公式看着复杂，拆开来看每个部分含义很直观：

- $f(t, d)$ 是词项 $t$ 在文档 $d$ 中出现的频率，它位于分子和分母中，体现了词频的非线性效应
- $k_1$ 控制着词频的饱和速度，通常取 1.2～2.0。当 $k_1 = 0$ 时词频完全不起作用，BM25 退化为 IDF 排序；当 $k_1$ 很大时词频接近线性增长
- $b$ 控制文档长度归一化的强度，取值在 $[0, 1]$，通常取 0.75。当 $b = 0$ 时完全不考虑文档长度，当 $b = 1$ 时完全归一化到平均长度
- $|d|$ 是当前文档长度，$\text{avgdl}$ 是所有文档的平均长度。比值 $|d| / \text{avgdl}$ 大于 1 时（长文档），分母中的长度项变大，词频得分被压缩，防止长文档仅因包含更多词就天然得分更高
- $\text{IDF}(t)$ 是逆文档频率，沿用斯帕克·琼斯 1972 年的核心思想：一个词如果出现在几乎所有文档中，它区分文档的能力就弱
- 最外层的 $\sum_{t \in q}$ 表示对查询中每个词项的得分求和

BM25 在精确匹配场景中至今没有真正的替代者。专有名词、产品编号、技术术语的检索几乎总是稀疏检索更优，这类场景中语义模糊只会带来噪声。SPLADE 等学习型稀疏检索方法通过神经网络学习"哪些词应该互相激活"，在保持稀疏索引高效性的同时获得了一定的语义泛化能力，是 BM25 的现代升级方向。

### 融合策略

现在有两路检索结果了：稠密检索返回余弦相似度（取值范围 $[-1, 1]$），稀疏检索返回 BM25 分数（取值范围 $[0, +\infty)$）。两路分数的量纲和分布完全不同，直接相加相当于把"考试成绩"和"身高"直接加权求和，毫无意义。融合策略要解决的就是这个跨路分数可比性的问题。

#### 加权融合

最直观的思路是先把两路分数缩放到同一个量纲上，再加权组合：

$$s_{\text{hybrid}} = \alpha \cdot s_{\text{dense}}' + (1 - \alpha) \cdot s_{\text{sparse}}'$$

分数归一化最常用的方法是 Min-Max 归一化：

$$s' = \frac{s - s_{\min}}{s_{\max} - s_{\min}}$$

它把任意区间的分数线性映射到 $[0, 1]$。举例来说，如果某次稠密检索返回的余弦相似度范围是 $[-0.1, 0.9]$，$-0.1$ 映射为 0，$0.9$ 映射为 1，中间的分数按比例缩放。但 Min-Max 的问题是对异常值敏感：如果某一个文档的分数极高（比如因巧合出现了 0.99 的余弦相似度），其他所有文档的归一化分数都会被压缩到很窄的区间，区分度丧失殆尽。分位数归一化（用 $P_{95}$ 替代 $s_{\max}$）能缓解这个问题，工程实践中更推荐。

权重 $\alpha$ 控制了稠密检索和稀疏检索的贡献比例。$\alpha$ 的调优通常依赖验证集上的网格搜索，但一个实用的经验是：事实型查询（如"2024 年诺贝尔物理学奖得主是谁"）$\alpha$ 宜小，偏重 BM25 的字面匹配；语义型查询（如"如何提高深度学习模型训练效率"）$\alpha$ 宜大，偏重向量的语义理解。更先进的方案是训练一个自适应调节模型，根据查询本身的特征预测最优权重，不过这套方案已经超出了本文的范围。

#### 倒数排名融合（RRF）

加权融合需要谨慎处理分数归一化，这本身就是一个容易出错的工程细节。倒数排名融合走了一条更简洁的路：完全不用原始分数，只看排名位置：

$$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)}$$

拆解来看：

- $\text{rank}_r(d)$ 是文档 $d$ 在第 $r$ 路检索结果中的排名，数字越小（越靠前）越好
- $k$ 是一个平滑参数，通常取 60。它防止排名第 1 和第 2 的文档之间差距过大，没有 $k$ 时，第 1 名得分是 1，第 2 名是 0.5，差了一倍；有 $k=60$ 时，第 1 名得分是 1/61 ≈ 0.0164，第 2 名是 1/62 ≈ 0.0161，差距被明显压缩
- 文档在某一路上排名越靠前，它的 RRF 得分就越高。文档在多个路上都排名靠前，RRF 得分会叠加

假设某文档在稠密检索中排第 1，在稀疏检索中排第 5，$k=60$：

$$\text{RRF} = \frac{1}{60 + 1} + \frac{1}{60 + 5} = 0.0164 + 0.0154 = 0.0318$$

RRF 的优势在于不依赖分数分布。无论稠密检索返回的余弦相似度分布在 $[-0.5, 0.8]$ 还是 $[0.3, 0.95]$，只要排名顺序相同，RRF 的结果就相同。这种对分数分布的不敏感性让 RRF 成为工程实践中最稳健的融合选择，这也是为什么 Elasticsearch 和许多开源 RAG 框架将 RRF 作为默认的混合检索策略。

| 策略 | 是否需要归一化 | 对分数分布敏感度 | 实现复杂度 |
|-----|-------------|----------------|-----------|
| 加权融合 | 是 | 高 | 低 |
| RRF | 否 | 低 | 低 |
| 学习型融合 | 否 | 低 | 高 |

## 重排序

混合检索通过组合两路检索思路来提升召回质量，但它本质上仍然是"用一个向量（或一个关键词）表示整个查询"的单次匹配。当候选文档集合大到成百上千时，这种压缩表示必然丢失信息。重排序的思路是承认初检的精度天花板，转而用一个更慢但更精准的模型对少量候选做二次筛选。

### 为什么需要重排序

初次检索（first-stage retrieval）受到两个约束。一是索引结构用了近似算法，IVF 只搜索少数聚类中心附近的区域，HNSW 按照贪心搜索图结构，两者都无法保证找到全局最近邻，召回率天然存在折损。二是双编码器架构将查询和文档分别独立编码为向量再计算相似度，查询中的"苹果"（手机）和文档中的"苹果"（水果）在向量空间中可能非常接近，因为双编码器无法在编码阶段看到对方的内容来做消歧。这两个约束加在一起，意味着初检结果中排在 Top-10 开外的相关文档会被漏掉，排在 Top-10 以内的不相关文档无法被甄别。

重排序的目标非常清晰：用更精确的模型对少量候选文档重新排序，实现"粗筛 + 精排"的两阶段架构。粗筛阶段用近似索引和双编码器在毫秒级别召回 100～200 篇候选，精排阶段用一个能捕获查询与文档交互的模型从中挑出最好的 5～10 篇。

```mermaid
graph LR
    A[用户查询] --> B[粗筛：混合检索<br/>双编码器<br/>返回 Top-100~200]
    B --> C[精排：交叉编码器 / LLM<br/>捕获细粒度交互<br/>返回 Top-5~10]
    C --> D[LLM 生成]
```

### 交叉编码器

双编码器（bi-encoder）的工作方式是把查询和文档分别喂给编码器、各自输出一个向量，然后在这两个向量的点积上判断相似度。这种"各算各的"方式的优点是文档向量可以提前预计算，检索时只需对查询做一次编码，速度快。缺点则很明显：编码查询时模型完全不知道文档内容，编码文档时也不知道查询意图，两条独立编码路径之间唯一的交汇点是最后的点积操作，这几乎不可能捕获"查询中的'快'是说速度还是说触感"与"文档中的'快'是说充电还是说快门"之间的微妙语义对应。

交叉编码器（cross-encoder）把查询和文档拼接成一个序列一起送入编码器，直接输出一个相关性分数。典型的拼接格式是 `[CLS] 查询 [SEP] 文档 [SEP]`，编码器（通常是 BERT 系的预训练模型）在自注意力层中让查询和文档的每个 token 都能互相看到，模型因此可以判断"查询中的'苹果'是指公司，文档中的'苹果'是指水果，不相关"。交叉编码器能捕获的交互模式包括：查询词在文档中是否出现了、出现的上下文是否符合同一个语义、查询与文档的核心主题是否一致，这些是双编码器架构在原理上就无法做到的。

代价是显而易见的：每次查询都需要将查询与每一篇候选文档拼接后重新编码，无法预计算。这也是为什么交叉编码器只用在精排阶段，候选集通常控制在 100～200 以内，在这个数量级上，交叉编码器的延迟尚可接受。常用的交叉编码器模型包括 BERT-base 在 MS MARCO 上微调的版本（如 `cross-encoder/ms-marco-MiniLM-L-6-v2`），以及更高效的蒸馏版本，后者可以在牺牲极少量精度的代价下将推理速度提升数倍。

### LLM 重排序

交叉编码器虽然精确，但针对特定领域需要微调，没有标注数据就无法发挥它的全部能力。LLM 重排序绕过训练环节，直接利用大语言模型的指令跟随能力：在提示词中给出查询和候选文档列表，要求 LLM 按相关性排序。提示词通常包括排序指令和输出格式（如"对以下文档按与查询的相关性排序，输出排序后的文档 ID 列表"）。

这种方法的泛化能力极强，无需任何领域内标注数据就能工作，而且 LLM 对文档内容的理解深度远超交叉编码器。但延迟和成本限制了大量使用：对 20 篇候选文档做排序可能需要数秒和数千 Token 的 API 消耗。适用于候选集很小（不超过 20 篇）且对精度有极高要求的场景，或者在缺乏标注数据时作为交叉编码器的替代方案。主流的实现之一是 RankGPT，它通过滑动窗口策略让 LLM 能处理稍大一些的候选集。

### 重排序方法对比

| 方法 | 精度 | 延迟 | 是否需要训练 | 适用候选集大小 |
|-----|------|------|------------|-------------|
| 双编码器 | 中 | 低 | 是 | 全量 |
| 交叉编码器 | 高 | 中 | 是 | 100-200 |
| LLM 重排序 | 高 | 高 | 否 | < 20 |

## 检索架构设计

将前面讨论的技术组合成可部署的检索系统，需要回答一个看似简单但直接影响成本和响应延迟的问题：需要几个检索阶段？每个阶段返回多少文档？

两阶段架构（粗筛 + 精排）是检索系统的标准设计。粗筛阶段的目标是高召回率，宁可多召回一些不相关的文档，也不能遗漏关键的。因此粗筛使用双编码器（文档向量可预计算）配合近似索引，在毫秒级别从海量文档中返回 Top-100～200。精排阶段的目标是高精确率，从 200 篇候选中挑出最能支撑生成的 5～10 篇。精排阶段用交叉编码器做逐对评分，候选集数量不大，所以准确率高而延迟可控。两阶段之间是一个自然的漏斗：粗筛负责"广撒网"，精排负责"精挑选"，架构简单、调优明确，是绝大多数 RAG 系统的选择。

当文档规模超过亿级时，两阶段架构可能不够用。200 篇候选对交叉编码器来说虽然延迟不高，但 200 篇中混入大量明显不相关的文档仍会浪费精排的计算资源。三阶段架构在粗筛和精排之间插入一个轻量级排序器：粗筛返回 Top-500～1000，粗排（通常用轻量级交叉编码器的蒸馏版本或加权的余弦相似度）压缩到 100～200，精排再用完整交叉编码器筛选到 10～20。每增加一个阶段，延迟增加但质量提升，最终阶段数量取决于业务对延迟的容忍预算。

工程实现上还有几个不容忽视的细节。稠密检索和稀疏检索是两支完全独立的检索管线，没有数据依赖，天然适合并行执行，两路检索同时发起，结果合并后统一去重。缓存策略有三个维度可以挖掘：查询缓存（完全相同的查询直接返回缓存结果，命中率取决于查询分布的集中程度）、文档编码缓存（稠密向量和稀疏倒排索引都可以提前计算并持久化）、重排序缓存（高频查询与候选文档的交叉编码器得分可以被缓存，避免重复推理）。增量更新方面，稀疏索引天然支持，倒排索引可以在线插入新文档而不需要重建全量索引；稠密索引则视选择的索引结构而定，HNSW 支持逐点插入，IVF 在聚类中心漂移时可能需要定期重建。

需要注意的是，上面讨论的架构都是批处理视角的设计。真实的生产系统中，冷启动、查询波动、索引更新频率、GPU 资源调度等问题往往比检索算法本身更难处理。这些工程问题不在本文讨论范围内，但在部署前需要纳入考量。

## RAG 端到端评估

检索评估只度量"检索到了什么"，但 RAG 系统的最终交付是生成给用户的文字回答，而不是 Top-K 文档列表。有时候检索指标漂亮但生成质量糟糕：系统找对了文档，但 LLM 的上下文整合能力有限，或者文档中的关键信息藏在冗长的中间段落里被模型忽略。端到端评估关注的不再是检索环节的得分，而是整个 RAG 管线的输出质量：回答有没有忠实于检索到的文档？有没有完整覆盖用户的提问？

端到端评估通常从三个维度展开。**忠实度**（Faithfulness）检查生成回答中的每一个陈述是否都能在检索到的文档中找到依据。如果文档中说的是"Transformer 发表于 2017 年"，回答中却说"Transformer 首次发表于 Google 的论文《Attention Is All You Need》"，这个陈述是忠实的，虽然措辞不同，但事实来源是文档中的信息。但如果回答中补充了"该论文获得了 NeurIPS 2017 最佳论文奖"，而文档中并未提及，这就是幻觉。**相关性**（Relevance）判断回答是否直接回答了用户的问题，即使回答完全忠实于文档，如果检索到的文档与查询不相关（精确率问题），生成的回答自然也文不对题。**完整性**（Completeness）评估回答是否覆盖了问题的各个方面：用户问的是"位置编码有哪些类型"，如果只提到了正余弦位置编码而遗漏了可学习编码和相对位置编码，完整性就受损。

检索不足（召回率低）和检索噪声（精确率低）对生成的影响模式截然不同。检索不足的典型症状是回答中出现"我不知道""没有相关信息"或者答非所问；检索噪声的表现是回答表面上看起来通顺，但细节与事实偏离，因为模型被不相关文档中的信息误导。检索数量 $k$ 的选择需要在这两种失败模式之间取得平衡：$k$ 过大引入噪声，$k$ 过小遗漏信息。RAG 场景中 $k = 3 \sim 5$ 是常见的起始值，这个范围在大多数嵌入模型的精度水平下能较好地兼顾召回率和噪声控制，但具体任务需要通过端到端评估验证。

人工标注三个维度的成本极高，一个 200 条查询的测试集就需要对 200 个回答逐一评判忠实度、相关性和完整性。RAGAS 等框架通过 LLM-as-Judge 模式来自动化这一流程：用大语言模型扮演"裁判"角色，按预定义的评分标准（例如"回答中的每句话是否都能在上下文中找到支撑"）打分量表。LLM-as-Judge 与人类标注的一致性在忠实度维度上已经相当高（通常达到 0.7 以上的相关性），可以作为快速迭代中的替代方案。但需要注意的是，LLM 评判器本身也有偏见，它倾向于给自己的"同类"（其他 LLM 生成的回答）打更高分，因此关键场景仍需人工抽检校准。

## 本章小结

从克兰菲尔德实验到 RAGAS 自动评估，检索评估的核心思路从未改变：用标准答案衡量检索质量，再依据评估结果针对性地优化系统。评估工具链覆盖了集合级的 Recall 和 Precision、排序级的 MRR/MAP/NDCG，以及端到端的忠实度/相关性/完整性评估，每个层次回答的问题各不相同但相互补充。优化手段包括混合检索（取稠密与稀疏之长，RRF 是工程实践中最稳健的融合策略）、重排序（用精度换效率，两阶段架构是标配），以及通过端到端反馈循环验证检索参数（如 $k$ 的取值）是否真的带来了生成质量的提升。

本章未涵盖的话题包括：不同嵌入模型对检索质量的影响（将在后续实验章节中通过基准评测来探讨）、检索结果的多样性指标（确保 Top-K 结果不只覆盖同一个角度的内容）、以及检索系统的在线评估方法（A/B 测试和用户行为信号分析）。此外，检索与生成的交互不仅仅发生在"检索结果喂给 LLM"这一个环节，更先进的 RAG 架构中，生成结果可以反向驱动检索，形成多轮检索 - 生成的闭环，这也是后续章节将要讨论的方向。

1. 实现检索评估流程，计算 Recall@k、Precision@k、MRR 和 NDCG@k，对比不同嵌入模型在同一数据集上的检索质量。

   <details>
   <summary>参考答案</summary>

   ```python runnable
   import numpy as np

   # 模拟检索结果：3 个查询，每个查询返回 5 篇文档的相关性判断（1=相关，0=不相关）
   # 每行对应一个文档：相关标记位于检索结果中的位置
   retrieved_relevance = [
       [1, 0, 1, 0, 0],  # 查询1：位置1和3相关
       [1, 1, 0, 0, 1],  # 查询2：位置1、2、5相关
       [1, 0, 1, 1, 0],  # 查询3：位置1、3、4相关
   ]
   total_relevant = [3, 4, 3]  # 每个查询的知识库中相关文档总数

   def recall_at_k(relevance, total_rel, k):
       return sum(relevance[:k]) / total_rel

   def precision_at_k(relevance, k):
       return sum(relevance[:k]) / k

   def mrr(relevance_list):
       rr_sum = 0
       for rel in relevance_list:
           for i, r in enumerate(rel):
               if r == 1:
                   rr_sum += 1 / (i + 1)
                   break
       return rr_sum / len(relevance_list)

   def ndcg_at_k(relevance, k):
       rel = relevance[:k]
       dcg = sum((2**r - 1) / np.log2(i + 2) for i, r in enumerate(rel))
       ideal = sorted(relevance, reverse=True)[:k]
       idcg = sum((2**r - 1) / np.log2(i + 2) for i, r in enumerate(ideal))
       return dcg / idcg if idcg > 0 else 0

   k = 5
   for i, rel in enumerate(retrieved_relevance):
       print(f"查询 {i+1}: Recall@{k}={recall_at_k(rel, total_relevant[i], k):.2f}, "
             f"Precision@{k}={precision_at_k(rel, k):.2f}, "
             f"NDCG@{k}={ndcg_at_k(rel, k):.2f}")

   print(f"\nMRR = {mrr(retrieved_relevance):.3f}")
   ```

2. 实现混合检索（稠密检索 + BM25），分别用加权融合和 RRF 合并两路结果。选择一个小型文档集（如 MS MARCO 的子集或自建的中文语料），对比两种融合策略在 Recall@5、MRR 和 NDCG@5 上的差异，并分析 RRF 在什么条件下显著优于加权融合。

   <details>
   <summary>参考答案</summary>

   ```python runnable
   import numpy as np

   # 模拟 10 篇文档的稠密检索分数（余弦相似度）和稀疏检索分数（BM25）
   dense_scores = np.array([0.92, 0.85, 0.78, 0.72, 0.65, 0.60, 0.55, 0.48, 0.40, 0.35])
   sparse_scores = np.array([15.2, 8.7, 22.1, 5.3, 18.9, 3.2, 7.8, 12.4, 2.1, 9.5])

   # 真实相关性标签（1=相关）
   relevance = np.array([1, 1, 1, 0, 0, 1, 0, 0, 1, 0])

   def min_max_norm(scores):
       return (scores - scores.min()) / (scores.max() - scores.min() + 1e-8)

   def weighted_fusion(dense, sparse, alpha=0.5):
       return alpha * min_max_norm(dense) + (1 - alpha) * min_max_norm(sparse)

   def rrf(dense_scores, sparse_scores, k=60):
       dense_rank = np.argsort(np.argsort(-dense_scores)) + 1
       sparse_rank = np.argsort(np.argsort(-sparse_scores)) + 1
       return 1/(k + dense_rank) + 1/(k + sparse_rank)

   wf_scores = weighted_fusion(dense_scores, sparse_scores)
   rrf_scores = rrf(dense_scores, sparse_scores)

   wf_order = np.argsort(-wf_scores)
   rrf_order = np.argsort(-rrf_scores)

   print(f"加权融合 Top-5: {wf_order[:5]}, 相关文档: {np.where(relevance[wf_order[:5]] == 1)[0]}")
   print(f"RRF Top-5:     {rrf_order[:5]}, 相关文档: {np.where(relevance[rrf_order[:5]] == 1)[0]}")
   ```
