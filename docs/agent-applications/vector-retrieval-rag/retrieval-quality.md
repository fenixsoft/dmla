# 检索质量：评估与优化

检索系统的价值取决于它能否找到用户真正需要的信息。前面章节介绍了向量索引结构与相似度计算，这些是检索的"基础设施"。但基础设施搭建完成后，如何判断检索效果好不好？效果不好时又该如何改进？本文回答两个问题：如何量化评估检索质量，以及如何通过混合检索与重排序提升检索质量——评估是度量的手段，优化是改进的方法，两者构成检索系统迭代的闭环。

## 检索评估指标

### 集合级指标：召回率与精确率

**召回率**（Recall）衡量"该找到的找到了多少"：

$$\text{Recall}@k = \frac{|\text{相关文档} \cap \text{检索结果前k}|}{|\text{相关文档}|}$$

在 RAG 场景中，召回率低意味着关键信息被遗漏，模型生成时缺少必要依据。

**精确率**（Precision）衡量"找到的有多少是正确的"：

$$\text{Precision}@k = \frac{|\text{相关文档} \cap \text{检索结果前k}|}{k}$$

精确率低意味着上下文窗口被无关信息污染，不仅浪费 Token，还可能误导生成。

两者存在天然矛盾：扩大搜索范围提升召回率，但不可避免引入噪声降低精确率。F1 分数取两者的调和平均，适合需要单一指标综合评价的场景：

$$F1 = \frac{2 \times \text{Precision} \times \text{Recall}}{\text{Precision} + \text{Recall}}$$

### 排序级指标：MRR、MAP、NDCG

集合级指标只关注"哪些文档被检索到"，忽略了排序位置的影响。实际上，排在前面的文档对 RAG 生成的影响远大于排在后面的——LLM 对上下文开头的段落更敏感（Lost in the Middle 问题）。排序级指标弥补了这一缺陷。

**MRR**（Mean Reciprocal Rank）关注第一个相关文档的位置：

$$\text{MRR} = \frac{1}{|Q|} \sum_{i=1}^{|Q|} \frac{1}{\text{rank}_i}$$

适用场景：用户只需要一个正确答案（如问答系统），第一个相关结果排在第 1 位得 1 分，排在第 3 位得 1/3 分。

**MAP**（Mean Average Precision）考虑所有相关文档的位置：

$$\text{AP} = \frac{1}{|\text{相关文档}|} \sum_{k=1}^{n} P(k) \times \text{rel}(k)$$

在每个相关文档出现的排序位置上计算精确率，再取平均。与 MRR 的区别：MAP 关注多个相关文档的整体排序质量。

**NDCG**（Normalized Discounted Cumulative Gain）支持多级相关性标注（不仅是"相关/不相关"的二值判断）：

$$\text{DCG}@k = \sum_{i=1}^{k} \frac{2^{\text{rel}_i} - 1}{\log_2(i + 1)}$$

$$\text{NDCG}@k = \frac{\text{DCG}@k}{\text{IDCG}@k}$$

折扣因子 $\frac{1}{\log_2(i+1)}$ 让排序靠后的文档权重递减，模拟用户注意力衰减。IDCG 是理想排序下的 DCG，用于归一化到 $[0, 1]$。

### 指标选择指南

| 指标 | 相关性标注 | 关注点 | 典型场景 |
|-----|-----------|-------|---------|
| Recall@k | 二值 | 覆盖度 | RAG 检索 |
| Precision@k | 二值 | 准确度 | 结果筛选 |
| MRR | 二值 | 首位准确 | 问答系统 |
| MAP | 二值 | 排序质量 | 文档检索 |
| NDCG@k | 多级 | 排序质量 | 搜索引擎 |

RAG 场景中最应关注 Recall@k——遗漏关键文档的代价远大于引入少量噪声。

### 评估数据集

指标计算需要"标准答案"：一组查询和对应的相关性标注。

- **查询集设计**：需覆盖不同类型（事实型、推理型、对比型）和不同难度
- **相关性标注**：二值标注（相关/不相关）简单但信息少，多级标注（高度相关/相关/弱相关/不相关）更精细但标注成本高
- **LLM 辅助标注**：用 GPT-4 等模型生成初始标注，人工审核修正，可大幅降低标注成本

常用评估基准：MS MARCO（段落检索）、BEIR（跨领域零样本评估）、C-MTEB（中文嵌入模型评估）。

## 混合检索

### 单一检索的局限

稠密检索与稀疏检索各有盲区。稠密检索语义泛化强——"汽车"能匹配"轿车"，但精确匹配差——"RFC 7231"可能匹配到"RFC 7230"。稀疏检索（如 BM25）恰好相反：精确匹配可靠，但"汽车"无法匹配"轿车"。混合检索的核心动机就是取长补短。

### 稀疏检索：BM25

BM25 是 TF-IDF 的改进，引入文档长度归一化：

$$\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t, d) \cdot (k_1 + 1)}{f(t, d) + k_1 \cdot (1 - b + b \cdot |d| / \text{avgdl})}$$

参数 $k_1$ 控制词频饱和度（词出现多少次后不再增加分数），$b$ 控制长度归一化强度。BM25 在精确匹配场景中不可替代——专有名词、编号、术语的检索几乎总是稀疏检索更优。

SPLADE 等学习型稀疏检索通过模型学习稀疏表示，在保持稀疏性的同时获得一定语义泛化能力，是传统 BM25 的升级方案。

### 融合策略

混合检索的关键问题：稠密检索返回余弦相似度（$[-1, 1]$），稀疏检索返回 BM25 分数（$[0, +\infty)$），两者不可直接比较。需要一种融合策略将两路结果合并为统一排序。

#### 加权融合

将两路分数归一化到同一区间后加权求和：

$$s_{\text{hybrid}} = \alpha \cdot s_{\text{dense}}' + (1 - \alpha) \cdot s_{\text{sparse}}'$$

归一化常用 Min-Max：$s' = \frac{s - s_{\min}}{s_{\max} - s_{\min}}$，但对异常值敏感，分位数归一化更稳健。权重 $\alpha$ 需根据查询类型调整——事实型查询偏稀疏，语义型查询偏稠密。

#### 倒数排名融合（RRF）

$$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)}$$

RRF 不需要分数归一化，直接使用排名位置。参数 $k$（通常取 60）平滑排名差异，避免排名第 1 和第 2 的分数差距过大。RRF 对分数分布不敏感，工程实现更稳健，是混合检索的默认选择。

| 策略 | 是否需要归一化 | 对分数分布敏感度 | 实现复杂度 |
|-----|-------------|----------------|-----------|
| 加权融合 | 是 | 高 | 低 |
| RRF | 否 | 低 | 低 |
| 学习型融合 | 否 | 低 | 高 |

## 重排序

### 为什么需要重排序

初次检索（First-stage Retrieval）追求效率，使用近似算法（IVF、HNSW）和双编码器（查询与文档分别编码后计算向量相似度）。效率的代价是精度——近似索引有召回损失，双编码器无法捕获查询与文档之间的细粒度交互。

重排序的核心思路：用更精确但更慢的模型对少量候选文档重新排序，实现"粗筛 + 精排"的两阶段架构。

```
查询 → 粗筛（混合检索，返回 Top-100）→ 精排（交叉编码器，返回 Top-10）→ 生成
```

### 交叉编码器

双编码器将查询和文档分别编码为向量再计算相似度，交叉编码器将查询和文档拼接后一起编码，直接输出相关性分数。交叉编码器能捕获查询与文档之间的细粒度交互——"这个词在查询和文档中是否表达了相同含义"，这是双编码器做不到的。

代价是每次查询都需要对所有候选文档重新编码，无法预计算。因此交叉编码器只用于精排阶段，候选集通常控制在 100-200 以内。

### LLM 重排序

使用大语言模型对候选文档重排序，通过提示词让 LLM 判断每个文档与查询的相关性。泛化能力强、无需训练，但延迟高、成本高，适用于候选集小（< 20）且对精度要求极高的场景。

### 重排序方法对比

| 方法 | 精度 | 延迟 | 是否需要训练 | 适用候选集大小 |
|-----|------|------|------------|-------------|
| 双编码器 | 中 | 低 | 是 | 全量 |
| 交叉编码器 | 高 | 中 | 是 | 100-200 |
| LLM 重排序 | 高 | 高 | 否 | < 20 |

## 检索架构设计

### 两阶段架构

粗筛阶段追求高召回率，允许低精确率，返回 Top-100；精排阶段追求高精确率，只处理少量候选，返回 Top-10。这是检索系统的标准设计，在召回率和延迟之间取得平衡。

### 多阶段架构

超大规模数据（> 1 亿文档）可采用三阶段架构：粗筛（Top-500）→ 粗排（轻量级交叉编码器，Top-100）→ 精排（重量级交叉编码器或 LLM，Top-10）。每增加一个阶段，延迟增加但质量提升，需根据业务延迟预算决定。

### 工程实现要点

- **并行检索**：稠密检索与稀疏检索并行执行，结果合并后去重
- **缓存策略**：查询缓存（相同查询直接返回）、文档编码缓存（预计算稠密和稀疏表示）、重排序缓存（高频查询-文档对）
- **增量更新**：稀疏索引天然支持（倒排索引），稠密索引需视类型而定（HNSW 支持插入，IVF 需重建）

## RAG 端到端评估

检索评估只度量"检索到了什么"，但 RAG 的最终目标是生成高质量回答。端到端评估关注检索质量对生成质量的影响。

### 评估维度

- **忠实度**（Faithfulness）：生成内容是否与检索到的文档一致，是否出现幻觉
- **相关性**（Relevance）：生成内容是否回答了用户的问题
- **完整性**（Completeness）：生成内容是否涵盖了问题的所有方面

### 检索与生成的联合分析

检索不足（召回率低）和检索噪声（精确率低）对生成的影响模式不同：检索不足导致回答不完整或"我不知道"，检索噪声导致回答偏离主题或产生幻觉。检索数量 $k$ 的选择需要权衡——$k$ 过大引入噪声，$k$ 过小遗漏信息，通常 $k=3 \sim 5$ 是 RAG 场景的常用值，但需根据具体任务验证。

RAGAS 等框架提供自动化的端到端评估方法，使用 LLM-as-Judge 替代人工标注，降低评估成本。

## 本章小结

评估与优化是检索系统迭代的闭环：先建立指标体系度量现状，再通过混合检索与重排序提升质量。Recall 和 Precision 衡量集合级质量，MRR/MAP/NDCG 衡量排序级质量；混合检索取稠密与稀疏之长，RRF 是最稳健的融合策略；重排序用精度换效率，两阶段架构是工程标配。RAG 场景中还需端到端评估，确认检索优化确实带来了生成质量的提升。
